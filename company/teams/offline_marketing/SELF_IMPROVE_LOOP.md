# 오프라인 마케팅팀 자기개선 루프

## 개요
리안이 피드백을 주지 않아도 팀 스스로 아웃풋을 점검하고 개선하는 구조.

---

## 동작 방식

### Phase 1: 파이프라인 실행 (기존)
```
researcher → strategist → copywriter → validator → 보고
```

### Phase 2: 자가점검 루프 (신규)
```
[아웃풋 생성] → post_run_critique() → [이슈 발견] → [재작성] → [재점검] → [통과/미해결]
```

**최대 2회 자가검사:**
- Iteration 1: 첫 점검 → 이슈 발견 → Sonnet으로 스크립트 수정
- Iteration 2: 재점검 → 통과 또는 미해결 이슈 기록

---

## 자가점검 5가지 기준

### 1. 가격 심리
- **점검**: 총액(6개월 합계)이 크게 노출되면 거절률 증가
- **정상**: 월액 기준으로 표현
- **문제**: 6개월 합계가 먼저 제시됨
- **개선**: 월액 강조 + 투자 대비 효과 강조

### 2. 해결책 노출
- **점검**: "이렇게 하세요" 류 표현 있으면 신뢰도 하락
- **정상**: 문제만 질문으로 유도
- **문제**: 구체적인 해결책 제시
- **개선**: 질문형으로 전환 (예: "요즘 신규 고객 확보 어떻게 하고 계세요?")

### 3. 손실 가시화
- **점검**: 월 손실금액이 투자금(DM비, 시간)과 비교 안 되면 설득력 약함
- **정상**: "월 ~명 잃고 있는데 그게 몇 백만원 손실"
- **문제**: 손실액이 모호하거나 추상적
- **개선**: 구체적 수치로 손실액 계산

### 4. CTA 명확도
- **점검**: 마지막에 연락처/카톡/전화 버튼 없으면 액션율 0
- **정상**: 명확한 행동 유도 (카톡 링크, 전화번호, 예약 링크)
- **문제**: "궁금하시면 연락주세요" 같은 모호한 표현
- **개선**: "오늘 카톡으로 무료 진단해드립니다 → [링크]"

### 5. 개인화
- **점검**: 업체명/지역이 없으면 스팸 분류 확률 증가
- **정상**: 업체명, 지역, 업종이 명시됨
- **문제**: 템플릿 방식 (모든 업체에 동일 메시지)
- **개선**: 변수화 가능한 구조 구성 ({업체명}, {지역}, {업종})

## 코드 구조

### post_run_critique() 함수
```python
def post_run_critique(output: dict, client: anthropic.Anthropic, max_iterations: int = 2) -> dict:
    """
    - 입력: copy, strategy, industry
    - 반복: 최대 2회
      1. Opus로 5가지 기준 점검
      2. 이슈 발견 → Sonnet으로 스크립트 수정
      3. 재점검
    - 출력: {passed, iterations, issues, improvements, final_copy}
    """
```

### pipeline.py 의 실행 순서
```python
# 기존
research → strategy → copy → validation

# 신규 추가
+ post_run_critique(output, client, max_iterations=2)
  → 자가점검 보고서 (_critique_report.json) 저장
  → 수정된 스크립트 저장 (영업_스크립트_v2_자가수정.md)

# 최종
+ 보고 (자가점검 결과 포함)
```

---

## 실행 흐름 (상세)

### 1단계: 첫 번째 자가점검
```
Claude Opus → 5가지 기준으로 현재 스크립트 평가
  ├─ 이슈 발견: {criterion, status, detail} 배열
  ├─ 개선 제안: [구체적 개선안 1, 2, 3]
  └─ has_issues: true/false
```

### 2단계: 이슈 있으면 스크립트 수정
```
if has_issues and iteration < 1:
  Claude Sonnet → 개선 제안을 반영해 스크립트 재작성
  → current_copy 업데이트
```

### 3단계: 재점검
```
if iteration == 1:
  Claude Opus → 수정된 스크립트 다시 평가
  → 이슈 없으면: passed = True (루프 탈출)
  → 이슈 있으면: unresolved_issues 기록
```

### 4단계: 보고
```
write_report() 호출 시:
  ✅ 자가점검 통과 (X회 검토, 발견 이슈 Y개, 적용 개선안 Z개)
  또는
  ⚠️ 자가점검 진행 중 (미해결 이슈 N개)
```

## 에러 처리

**try/except로 감싸서 파이프라인 중단 방지:**
```python
try:
    critique_result = post_run_critique(...)
except Exception as e:
    print(f"⚠️  자가점검 루프 에러: {e}")
    context["critique_result"] = {"passed": False, "error": str(e)}
```

즉, 자가점검이 실패해도 기본 파이프라인은 계속 진행 → 리안에게 보고됨.

## 리안이 보는 보고서

### 자가점검 성공한 경우
```
✅ 자가점검 통과 (총 2회 검토)
- 발견 이슈: 3개
- 적용 개선안: 3개
```

### 자가점검 진행 중인 경우
```
⚠️ 자가점검 진행 중 (총 2회 검토)
- 미해결 이슈: 2개
- 상세: `_critique_report.json` 참조
```

→ 리안이 필요 시 `_critique_report.json` 확인 → "이 이슈 이렇게 수정해줘" 요청

## 핵심 원칙

1. **에러해도 멈추지 않기**: try/except
2. **최대 2회까지만**: 무한 루프 방지
3. **Opus는 점검만, Sonnet은 수정만**: 역할 분리
4. **구체적 개선안**: "어떻게 하세요?" 금지
5. **리안은 최종 판정만**: 중간 판정은 팀이
