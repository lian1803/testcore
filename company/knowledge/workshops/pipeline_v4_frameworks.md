# 🏗️ AI-Powered Lean Startup Pipeline — 전체 구조
1. 기능은 줄이고, '해결'은 완벽하게
특히 바쁜 소상공인이나 온라인 판매자를 위한 자동화 툴, 마케팅 SaaS 같은 B2B 서비스를 기획할 때 이 원칙이 절대적입니다. 이들은 복잡한 매뉴얼을 읽을 시간이 없습니다.
10가지 기능을 어설프게 제공하는 것보다, 고객의 가장 큰 골칫거리 단 하나를 클릭 한두 번 만에 시원하게 해결해 주는 것이 훨씬 더 큰 감동을 줍니다.

2. '아하 모먼트(Aha Moment)'에 모든 것을 집중
사용자가 우리 제품의 가치를 처음으로 깨닫는 순간(Aha Moment)까지 가는 길을 극단적으로 짧고 매끄럽게 깎아야 합니다.

"이 버튼을 누르면 정말 알아서 다 해준다고?"

"결과물이 이렇게 깔끔하게 나온다고?"
이런 반응을 끌어낼 수 있는 핵심 기능 딱 하나에 디자인, UX, 개발 리소스를 몰아넣는 것이 진짜 MLP입니다.

3. '예쁜 쓰레기'가 아닌 '친절한 전문가'
여기서 말하는 UX/UI의 강화는 단순히 겉모습을 화려하게 꾸미는 것이 아닙니다. 오류가 났을 때 친절하게 해결책을 알려주는 안내 문구, 로딩되는 동안 지루하지 않게 보여주는 매끄러운 애니메이션, 직관적인 버튼 위치 등 사용자를 배려하는 디테일을 의미합니다.

## NotebookLM: 상시 컨텍스트 허브

NotebookLM은 특정 단계 전용이 아니라, **모든 산출물을 누적 업로드하는 프로젝트 기억 저장소**다.

### 업로드 타이밍
- 1단계 끝 → `painpoints.json`
- 2단계 끝 → `reverse_engineering.md`
- 3단계 끝 → `strategy_brief.md`
- 4단계 끝 → `business_model.md`
- 5단계 끝 → `mvp_spec.md` + `agent_plan.md`
- 5.5단계 끝 → `tech_scout.md`
- 6단계 끝 → `ui_blueprint.md`
- 7단계 끝 → `prd.md` + `feature_specs.md` + `test_protocol.md`
- 9단계 끝 → `/ops/` 문서들
- 10단계 끝 → `growth_sprint_{N}.md`

### 활용 시점
- 단계 간 맥락이 끊길 때: "이전에 1순위로 뭘 골랐고 왜?"
- 설계와 구현이 어긋날 때: "mvp_spec에서 이 기능 어떻게 정의했어?"
- 외부 서비스 비용/대안 확인할 때: "tech_scout에서 OCR 후보가 뭐였지?"
- 10단계 반복 시: 누적된 데이터로 트렌드 분석

## 📌 2단계 프롬프트 (복붙용)

```
[입력 데이터]
아래(또는 첨부)는 {DOMAIN} 시장의 Pain Point 50개 JSON이다.

{painpoints.json 내용 붙여넣기 또는 파일 첨부}

---

[역할 정의]
당신은 '서비스 역기획 전문가'다.

당신의 임무:
위 JSON의 competitor_url 필드에서 가장 빈번하게 등장하거나
pain_level 4~5 항목에 연결된 경쟁 서비스 상위 3~5개를 선별하고,
각 서비스를 5-Layer로 역기획 분석하는 것이다.

---

[0] 역기획 대상 선정

JSON에서 아래 기준으로 분석 대상을 선정:
- competitor_url 등장 빈도 상위
- pain_level 4~5와 연결된 서비스 우선
- 직접 경쟁자 + 간접 경쟁자(다른 방식으로 같은 JTBD를 해결하는 서비스) 모두 포함
- 최소 3개, 최대 5개

각 대상에 대해 아래 [1]~[5]를 수행하라.

---

[1] Layer 1 — 정보 구조 (IA: Information Architecture)

해당 서비스의 화면/메뉴 구조를 트리로 재구성:
- 메인 네비게이션 구조
- 핵심 기능의 깊이 (몇 번 클릭해야 도달하는가?)
- 사용자가 가장 자주 쓸 것으로 추정되는 경로

---

[2] Layer 2 — 사용자 흐름 (User Flow)

핵심 사용자 여정을 분석:
- 가입 → 핵심 기능 사용 → 결제까지의 경로
- 단계 수 (많을수록 이탈 위험)
- 아하 모먼트 추정 (사용자가 "이거다!" 느끼는 시점)
- 이탈 가능 지점 (단계가 길거나, 입력이 많거나, 가치를 못 느끼는 곳)

---

[3] Layer 3 — UI/UX 분석

- 첫 화면에서 3초 안에 "이 서비스가 뭔지" 파악 가능한가?
- CTA(핵심 행동 유도)가 명확한가?
- 빈 상태/에러 상태/로딩 처리가 되어 있는가?
- 모바일 대응 여부

---

[4] Layer 4 — 비즈니스 모델 분석

- 수익 모델: 구독/건당/수수료/광고/프리미엄 등
- 가격 정책: Free Tier 범위, 유료 전환 트리거, 가격대
- 가격 앵커 전략 사용 여부
- 추정 TAM/SAM (이 서비스가 노리는 시장 규모)

---

[5] Layer 5 — 약점 + 기회 도출

JSON의 evidence_quote와 competitor_weakness를 교차 참조하여:
- 이 서비스에 대한 실제 사용자 불만 TOP 3
- 불만의 근본 원인 (기능 부족? UX 문제? 가격? 특정 세그먼트 무시?)
- 이 약점을 공략할 수 있는 구체적 기회 1~2개

### Value-Add 전략 질문 (각 경쟁사에 적용)
아래 질문으로 개선 기회를 체계적으로 도출:
- "이 서비스가 하지 않는 것은 무엇인가?" (결핍 기회)
- "이것을 10배 빠르게/쉽게 만들 수 있는가?" (효율 기회)
- "누가 이 서비스를 쓰다가 포기했을까?" (이탈자 기회)
- "이 서비스의 기존 고객 중 가장 불만인 세그먼트는?" (세분화 기회)
- "이 서비스에 없는 기능을 사용자가 직접 만들어 쓰고 있는가?" (Workaround 기회)

### Customer Forces Canvas (전환의 4가지 힘)
경쟁사 사용자가 "우리 서비스로 전환"하려면 4가지 힘의 합이 맞아야 한다.
각 경쟁사에 대해 아래를 분석:

| 힘 | 질문 | 분석 |
|---|---|---|
| **Push** (현 상황 불만) | 경쟁사 사용자가 참을 수 없는 불만은? | |
| **Pull** (새 솔루션 매력) | 우리가 제공할 "아, 이런 게 있었어?" 순간은? | |
| **Inertia** (관성·전환비용) | 사용자가 안 바꾸는 이유는? (데이터 이전, 학습 비용, 팀 설득 등) | |
| **Anxiety** (불안감) | 새 서비스에 대한 두려움은? (안정성, 데이터 유실 등) | |

**전환 조건: (Push + Pull) > (Inertia + Anxiety)**
→ Inertia와 Anxiety를 낮추는 전략이 3단계 아이디어 도출의 핵심 근거가 된다.

# 🔥 5.5단계 — Perplexity → GPT / Gemini / Claude (Tech Stack Scout)

---

## 이 단계가 필요한 이유

5단계는 "무엇을 만들지"를 정한다.
하지만 "무엇으로 만들지"는 빠져 있다.

5단계의 [9] 기술 스택은 프레임워크·DB·배포 같은 **기반 인프라**만 다룬다.
실제 MVP의 핵심 가치를 만드는 건 **외부 API, LLM, 써드파티 서비스**인 경우가 대부분이다.

예시:
- OCR이 필요한데 Tesseract? Google Vision? Upstage? → 정확도·가격·한국어 지원이 천차만별
- LLM이 필요한데 GPT-4o? Claude Sonnet? Gemini Flash? 로컬 모델? → 용도별 최적해가 다름
- 이메일 발송에 SendGrid? Resend? AWS SES? → 프리 티어·가격·API 편의성이 다름
- "나노바나나"처럼 내가 존재조차 모르는 최신 서비스가 더 나은 선택일 수 있음

이 단계를 건너뛰면:
- 개발 중 "이거 API 뭐 쓰지?" 하고 멈추게 됨
- 감으로 고른 서비스가 비싸거나 정확도가 낮아서 나중에 교체해야 함
- 더 좋은 대안이 있는 줄 모르고 바퀴를 재발명함
- 8~9단계에서 AI Coder가 아무 API나 골라버림

### [TP-2] 테스트 스크립트

#### 인트로 (3분)
"안녕하세요. 오늘 저희 서비스의 사용성을 테스트하려고 합니다.
정답은 없고, 서비스가 테스트 받는 거지 참가자분이 테스트 받는 게 아닙니다.
불편하거나 헷갈리는 점을 솔직하게 말씀해주시면 큰 도움이 됩니다.
사용하시면서 생각나는 것을 소리 내어 말씀해주세요 (Think Aloud)."

#### 워밍업 질문 (3분)
1. [도메인] 관련해서 평소 어떤 서비스를 사용하시나요?
2. 그 서비스에서 가장 불편한 점은 뭔가요?
3. 이런 문제를 해결하기 위해 시도해본 것이 있나요?

#### 핵심 태스크 (20분)

mvp_spec.md의 Must 기능 기반으로 3~5개 태스크 설계.
각 태스크마다:

| 태스크 | 시나리오 | 성공 기준 | 관찰 포인트 |
|--------|---------|----------|-----------|
| T1 | "당신은 [상황]입니다. [서비스]를 사용해서 [목표]를 달성해보세요." | 완료 여부 + 소요 시간 | 멈추는 곳, 헤매는 곳, 잘못 누르는 곳, 예상과 다른 행동 |
| T2 | | | |
| T3 | | | |

#### 진행 중 프롬프트
관찰하면서 아래 질문을 적절히 사용:
- "지금 무슨 생각을 하고 계세요?"
- "이 버튼을 누르면 뭐가 나올 거라고 기대하세요?"

### [TP-3] 피드백 분석 프레임워크

테스트 완료 후 수집된 피드백을 아래 형식으로 정리:

| 피드백 | 분류 | 심각도 | 빈도 (5명 중) | 영향 퍼널 | 개선안 | 난이도 | 우선순위 점수 |
|--------|------|--------|-------------|----------|--------|--------|-------------|
| | UI/UX/성능/기능 | Critical/Major/Minor | | AARRR 중 | | 상/중/하 | |

**우선순위 공식**: Impact(1~5) × Frequency(1~5) ÷ Effort(1~5) = 점수
- 점수 높을수록 먼저 수정
- 2명 이상 동일 문제 = 반드시 수정 (Must Fix)
- Quick Win (높은 Impact, 낮은 Effort) 별도 표시

### 🚦 사전 검증 게이트: Demo-Sell-Build

> "대부분의 창업자는 Build → Demo → Sell 순서로 한다. 뒤집어라." — Ash Maurya

**7단계 산출물이 완성되면, 코딩(8단계) 전에 반드시 이 순서를 거친다:**

| 순서 | 행동 | 구체적 방법 | 소요 |
|------|------|------------|------|
| **① Demo** | PRD/와이어프레임/목업을 잠재 고객에게 보여준다 | Figma 목업, AI 생성 스크린샷, 또는 말로 설명도 데모다 | 1일 |
| **② Sell** | "이 서비스가 월 X원이면 쓰시겠어요?" 반응을 확인한다 | **약속**을 받아야 한다: 사전결제, 이메일 등록, 다음 미팅 예약 (Mom Test 규칙 적용) | 3일 |
| **③ Build** | Sell에서 약속을 받은 후에만 8단계로 진행한다 | 약속 0건이면 → 3단계로 돌아가 아이디어/솔루션 재검토 | - |

**Go/No-Go 기준:**
- **Go** → 10명 중 3명 이상이 구체적 약속 (결제, 이메일, 지인 소개)
- **Pivot** → 0~2명만 약속 → Problem Interview 재실행 → 린 캔버스 수정 → 3단계 재진입

이 게이트를 건너뛰고 8단계로 가면, "아무도 안 쓰는 완벽한 앱"을 만드는 가장 빠른 길이다.

[0] 데이터 신뢰도 + VoC 분류

### 정량 데이터 신뢰도
- 표본 수가 통계적으로 유의미한지 판별
- 부족하면 "가설 단계" 표기 + 검증에 필요한 최소 표본 수 제시
- 충분하면 "유의미" 표기

### VoC (고객의 소리) 분류

| 분류 | 정의 | 긴급도 |
|------|------|--------|
| 버그(Critical) | 기능이 작동하지 않음 | 즉시 |
| UX 마찰(Friction) | 기능은 되지만 어렵거나 헷갈림 | 높음 |
| 결핍(Gap) | 이 기능이 없어서 이탈/미결제 | 핵심 |

각 분류별: 건수 + 대표 원문 인용 + 영향받는 퍼널 단계

### NPS 측정 (데이터가 있을 경우)
- Promoter(9~10) / Passive(7~8) / Detractor(0~6) 비율
- NPS 점수 = Promoter% - Detractor%
- Detractor의 주요 불만 사유 TOP 3
- NPS가 없으면 피드백 감정 분석으로 대체 추정 + "NPS 미측정 — 다음 스프린트에서 도입 권고"

[3] PLC (서비스 생애주기) 단계 판단

현재 서비스가 어느 단계에 있는가?

| 단계 | 특징 | 전략 |
|------|------|------|
| 도입기 | 인지도 낮음, 사용자 적음, 수익 없거나 미미 | → PSF 확인, 얼리어답터 집중, 빠른 피드백 루프 |
| 성장기 | 사용자 증가, 수익 발생, 경쟁자 진입 시작 | → PMF 강화, 채널 확장, 차별화 강화 |
| 성숙기 | 성장 둔화, 수익 안정, 경쟁 심화 | → 리텐션 최적화, 신규 세그먼트, 가격 실험 |
| 쇠퇴기 | 사용자 감소, 수익 하락 | → 피벗 또는 철수 |

현재 단계 판정 + 근거 + 해당 단계에 맞는 전략 방향.

[4] Kingpin 액션 — North Star를 가장 크게 움직이는 1개

[0]~[3] 종합하여 1순위 액션 선택:

| 항목 | 내용 |
|------|------|
| Action | 무엇을 수정하는가? (파일명/화면명 수준으로 구체적) |
| Flywheel 연결 | business_model.md의 Flywheel 어느 고리를 가속하는가? |
| Engine of Growth 정합 | Sticky/Viral/Paid 중 선택한 엔진에 부합하는가? |
| Target Cohort | 누구에게 먼저? |
| North Star 임팩트 | 얼마나 개선되는가? |
| 성공 기준 (Go) | 이 수치 이상이면 성공 |
| 실패 기준 (Stop Rule) | 이 수치 이하면 중단 → 2순위 |
| OKR 연결 | 어떤 KR에 기여? |

근거는 반드시 로그/피드백 원문 인용.
"마케팅 강화", "UX 개선" 같은 추상적 액션 금지.

[8] 피벗 판단 (필요 시)

Kingpin 실패 + Stop Rule 도달 시:

### 피벗 필요 여부 체크
- 48시간 검증(3단계) 결과는 긍정적이었는가?
- NPS Detractor 사유가 "문제 자체가 아님"인가, "솔루션이 안 맞음"인가?
- Retention 곡선이 0으로 수렴하는가?

### 피벗 유형 (해당되는 것 선택)
- Zoom-in: 하나의 기능만 독립 서비스로
- Zoom-out: 현재 서비스를 더 큰 서비스의 기능으로
- Customer Segment: 같은 제품, 다른 고객
- Customer Need: 같은 고객, 다른 문제
- Channel: 같은 제품, 다른 유통 채널
- Revenue Model: 같은 제품, 다른 수익 모델
- Value Capture: 가격/과금 방식 변경
- Engine of Growth: Sticky → Viral 등 성장 엔진 전환
- Technology: 같은 문제, 다른 기술
- Platform: 앱 → 마켓플레이스 등

피벗 시 → 파이프라인 어디로 돌아가는지 결정.
