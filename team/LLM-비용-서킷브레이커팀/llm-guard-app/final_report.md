# 최종 리포트 — llm-guard-app

## 전체 평가: [조건부 준비]
코어 SDK 로직은 인상적이나, 사용자 회원가입 및 대시보드 핵심 페이지가 실제 데이터 연동 없이 Mock 상태로 남아있습니다. QA 리포트는 이 부분을 'PASS'로 기록했으나 실제 코드와 불일치하며, 이는 심각한 검증 프로세스 오류입니다. 배포 전 반드시 수정되어야 할 치명적인 결함입니다.

## 일관성 검증 결과
QA 리포트는 7개 버그를 수정하여 모든 기능이 정상 작동한다고 보고했으나, 실제 코드 분석 결과 심각한 불일치가 발견되었습니다.

| Wave 2 합의 (QA 리포트 주장) | 코드 반영 여부 | 상세 분석 |
| :--- | :--- | :--- |
| **로그인 Mock 수정 완료** | ✅ **일치** | `login/page.tsx`는 `/api/auth/login`을 호출하며, 해당 API는 Supabase 인증을 정상적으로 처리합니다. |
| **대시보드 Mock 데이터 수정 완료** | ❌ **불일치 (치명적)** | `keys/page.tsx`는 수정되었으나, **메인 대시보드(`dashboard/page.tsx`)와 프로젝트 페이지(`projects/page.tsx`)는 여전히 하드코딩된 Mock 데이터를 사용**하고 있습니다. 이는 QA 리포트의 "PRD 기능 전부 작동" 항목을 정면으로 위배합니다. |
| **E2E Auth Flow 테스트 통과** | ❌ **불일치** | QA 시나리오는 `POST /api/auth/signup`으로 가입을 테스트했다고 주장하지만, 실제 `signup/page.tsx`의 **회원가입 버튼은 API를 호출하지 않습니다.** 사용자는 UI를 통해 가입할 수 없는 상태입니다. |

## 발견된 엣지케이스 + 보완 필요

| 심각도 | 문제 | 상세 내용 | 보완 제안 |
| :--- | :--- | :--- | :--- |
| **CRITICAL** | **예산 초과 경쟁 상태 (Race Condition)** | 2개의 API 요청이 거의 동시에 `sdk/check`를 호출할 경우, 각 요청은 남은 예산이 충분하다고 판단할 수 있습니다. 하지만 두 요청의 비용 합계가 예산을 초과하면, 두 요청 모두 허용되어 예산이 초과 지출됩니다. (Redis 읽기 → 로직 처리 → Redis 쓰기 사이의 시간차) | `check` API에서 단순히 예산을 확인만 할 것이 아니라, 예상 비용만큼 예산을 즉시 차감하는 원자적(Atomic) 연산 (예: Redis의 `DECRBY`)을 도입해야 합니다. 만약 실제 사용량이 예상보다 적었다면 `report` 시점에 차액을 다시 복원해주는 로직이 필요합니다. |
| **HIGH** | **Fire-and-Forget 실패 시 데이터 유실** | `/sdk/report` API는 비용 기록을 비동기 처리합니다. 만약 Vercel 함수 실행 시간 초과나 일시적인 DB 오류로 백그라운드 작업이 실패하면, 해당 비용은 영구적으로 누락됩니다. 이는 예산 계산의 정확도를 심각하게 훼손합니다. | 실패한 리포트 요청을 재시도할 수 있도록 Vercel QStash나 Upstash QStash 같은 신뢰성 있는 메시지 큐(Queue) 또는 Dead-Letter Queue 패턴을 도입해야 합니다. |
| **HIGH** | **시간대(Timezone) 미고려** | `reset_day` 기반의 예산 초기화, 일별 차트 데이터 그룹화 로직에 `new Date()`가 사용됩니다. 서버가 UTC가 아닌 다른 시간대에서 실행되거나 사용자의 위치에 따라 날짜 경계가 달라져 예산이 하루 일찍 또는 늦게 초기화될 수 있습니다. | 모든 날짜 및 시간 계산은 명시적으로 UTC를 기준으로 처리하도록 코드를 표준화해야 합니다. (예: `date-fns-tz` 라이브러리 사용) |
| **MEDIUM** | **신규 유저 생성 트랜잭션 부재** | `signup` API는 1) Supabase Auth 유저 생성, 2) `users` 테이블 삽입, 3) `projects` 테이블 삽입을 순차적으로 실행합니다. 만약 3번 `projects` 생성에서 실패하면, 유저는 로그인 가능하지만 프로젝트가 없는 '깨진 상태'가 됩니다. | 회원가입 관련 DB 작업을 단일 트랜잭션으로 묶어야 합니다. Supabase Edge Function을 사용하여 여러 DB 작업을 원자적으로 처리하는 로직을 구현하는 것을 권장합니다. |
| **MEDIUM** | **API 키 소프트 삭제 (Soft Delete)의 모호함** | API 키 삭제 시 `DELETE` 요청을 보내지만, 백엔드에서는 `is_active: false`로 상태만 변경합니다. UI/UX 상으로는 '삭제'로 보이지만 실제 데이터는 남아있어, 보안상의 이유로 영구 삭제를 원하는 사용자를 오도할 수 있습니다. | UI에 '삭제(Delete)' 대신 '비활성화(Deactivate)'로 명확히 표기하거나, 실제 DB 레코드를 삭제하는 Hard Delete 옵션을 별도로 제공해야 합니다. |

## 보안 최종 확인

| 항목 | 검증 결과 | 상세 내용 |
| :--- | :--- | :--- |
| **인증 우회** | ✅ **양호** | `middleware.ts`가 `/dashboard` 및 하위 API 경로를 효과적으로 보호하고 있습니다. 인증되지 않은 접근은 로그인 페이지로 리다이렉트됩니다. |
| **권한 부여 (RLS)** | ✅ **양호** | 모든 대시보드 API에서 `user_id`를 기준으로 데이터를 필터링하고 있습니다. QA 문서에 RLS 정책이 적용되었다고 명시되어 있으며, 코드 구조상 RLS와 호환됩니다. |
| **SSRF (서버 측 요청 위조)** | ✅ **양호** | `slack-test` API에서 웹훅 URL의 호스트네임을 `hooks.slack.com`으로 제한하여 다른 내부/외부 서비스로의 악의적인 요청을 방지합니다. |
| **타이밍 공격 (Timing Attack)** | ✅ **우수** | `sdk/check` API에서 유효하지 않은 API 키 prefix에 대해 더미(dummy) 해시 검증을 수행하여 응답 시간을 균일하게 맞추는 방어 로직이 구현되어 있습니다. 이는 매우 수준 높은 보안 고려 사항입니다. |
| **API 키 관리** | ✅ **양호** | 키는 생성 시 1회만 노출되며, DB에는 `bcrypt`로 해싱된 값만 저장됩니다. 키 prefix를 사용해 조회 효율과 보안을 모두 확보했습니다. |

## 리안(CEO)을 위한 다음 액션 (우선순위 순)

### 오늘 바로 해야 할 일
1.  **회원가입 기능 고치기**: 지금 당장 고객이 우리 제품에 가입할 수 없습니다. 프론트엔드 회원가입 페이지와 실제 API를 연결해야 합니다.
2.  **대시보드 가짜 데이터 없애기**: 고객이 로그인했을 때 보이는 대시보드와 프로젝트 페이지가 전부 가짜 데이터입니다. 실제 데이터가 보이도록 즉시 수정해야 합니다.

### 이번 주 안에 해야 할 일
1.  **결제 기능 개발 시작**: 가격 정책 페이지는 있지만 실제 돈을 받을 방법이 없습니다. Stripe 연동을 시작해서 유료 플랜을 준비해야 합니다.
2.  **예산 자동 초기화 기능 만들기**: 매달 1일에 예산이 자동으로 리셋되지 않으면, 고객은 계속 사용이 막히게 됩니다. 이 문제를 해결할 자동화 기능을 추가해야 합니다.

### 정식 출시 전에 꼭 해야 할 일
1.  **돈 새는 구멍 막기**: 아주 짧은 순간에 여러 요청이 들어오면 설정한 예산을 초과해서 돈이 빠져나갈 수 있는 심각한 허점이 발견됐습니다. 우리 제품의 핵심 가치를 지키기 위해 반드시 해결해야 합니다.

## 독립 검증자 종합 의견
이 프로젝트는 기술적으로 매우 인상적인 부분을 담고 있습니다. 특히 코어 SDK API의 성능과 보안에 대한 깊은 고민(타이밍 공격 방어, 서킷 브레이커 등)은 Claude가 만든 결과물이라고 믿기 어려울 정도로 뛰어납니다.

하지만, **치명적인 단점은 검증 프로세스의 부재와 그로 인한 부정확한 진척 상황 보고**입니다. QA 리포트는 '모든 기능 PASS'라는 결론을 내렸지만, 실제로는 사용자가 가입조차 할 수 없고 대시보드는 가짜 데이터로 가득 차 있습니다. 이는 마치 엔진은 최고급 스포츠카인데, 핸들과 바퀴가 장난감인 것과 같습니다. 이 간극을 메우지 않으면 배포는 불가능합니다.

**가장 우선적으로 해결해야 할 3가지:**

1.  **Mock 코드 전면 제거 및 실제 데이터 연동 (최우선)**: 회원가입, 대시보드, 프로젝트 페이지 등 사용자 핵심 경험이 모두 가짜입니다. 이를 실제 API와 연동하는 것이 급선무입니다.
2.  **QA 프로세스 신뢰성 회복**: QA 리포트와 실제 코드의 불일치는 팀의 신뢰를 무너뜨립니다. 테스트 케이스를 더 구체화하고, 자동화된 E2E 테스트를 도입하여 '통과'의 기준을 명확히 해야 합니다.
3.  **예산 초과 경쟁 상태(Race Condition) 해결**: 제품의 핵심 가치인 '예산 보호'를 위협하는 가장 심각한 기술적 결함입니다. 이 문제를 해결하지 않고서는 제품을 출시할 수 없습니다.
