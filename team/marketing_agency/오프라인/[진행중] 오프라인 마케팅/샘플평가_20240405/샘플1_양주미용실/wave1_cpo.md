# CPO 분석 -- 소상공인 네이버 플레이스 영업툴 업그레이드

## 제품 전략
리안이 콜드DM만으로 월 5건 계약을 달성하기 위한 "진단-제안-클로징" 자동화 엔진.
핵심은 진단 데이터를 영업 무기(PPT + 메시지)로 변환하는 속도와 정확도다.

---

## 타겟 + Pain
- **타겟**: 리안 (1인 영업자). 최종 고객은 경기북부 소상공인이지만, 이 툴의 사용자는 리안 자신이다.
- **Pain**: 업체 하나 진단하고 메시지 작성하는 데 시간이 너무 걸린다. 하루 30~50건 DM을 보내려면 진단-메시지 생성이 자동이어야 한다. 지금은 진단은 되지만 메시지는 수동 복붙, PPT도 영업용으로는 약하다.
- **아하 모먼트**: 업체명 하나 넣으면 30초 안에 진단 완료 + 1~4차 메시지가 복붙 가능한 상태로 화면에 뜨는 순간.

---

## 현재 코드 진단 (기존 자산 평가)

### 잘 되어 있는 것
1. **크롤링 파이프라인 완성**: 검색 -> 플레이스 상세 크롤링 -> DB 저장 -> PPT 생성 플로우가 돌아간다
2. **7개 항목 점수 체계**: photo/review/blog/info/keyword/convenience/engagement 분류와 가중치가 합리적
3. **DB 스키마 확장성**: 편의기능, 외부채널, 메뉴상세 등 세부 필드가 이미 models.py에 정의되어 있다
4. **개선 포인트 자동 생성**: scorer.py의 generate_improvement_points()가 우선순위 기반으로 최대 5개 추출

### 문제점 (영업 관점)
1. **점수가 절대값 기반**: 사진 15장이면 100점인데, 경쟁사가 50장이면 의미가 없다. 상대점수 개념이 없다
2. **PPT가 정보 전달용**: "사장님 지금 N명 놓치고 있어요" 같은 충격 메시지가 없다. 영업 제안서가 아니라 리포트 수준
3. **메시지 생성 기능 없음**: 1~4차 퍼널 메시지가 영업_퍼널_설계.md에 템플릿으로만 존재. 코드에 녹아있지 않다
4. **배치 처리 없음**: 업체 하나씩만 진단 가능. xlsx에서 목록 불러와서 일괄 처리 불가
5. **영업 우선순위 분류 없음**: 진단 후 1순위/2순위/패스 자동 태그가 없다

## MVP 범위 (Must Have) -- 이번에 반드시 해야 하는 것

### 1. 영업 메시지 자동 생성 (최우선)
> **이유**: 이게 없으면 진단 결과가 나와도 리안이 수동으로 숫자 바꿔가며 메시지 작성해야 한다. 하루 30건 DM이 불가능하다.

- 진단 완료 후 결과 화면에 1~4차 메시지 탭 추가
- 1차 메시지: 업체 데이터 기반으로 A(리뷰격차)/B(손실고객)/C(방치) 버전 자동 선택 + 생성
- 2차 메시지: 진단 데이터 -> 카톡 카드 형식 텍스트 자동 완성
- 3차 메시지: 업종별 객단가 기반 손익분기 계산 + 패키지 추천 자동 주입
- 4차 메시지: 보류/비싸다/직접하겠다 3가지 버전 자동 생성
- **각 메시지에 "복사" 버튼**: 클릭하면 클립보드 복사. 바로 카톡에 붙여넣기 가능

**구현 위치**: `services/message_generator.py` (신규) + `routers/message.py` (신규) + 결과 화면 UI 확장

### 2. PPT 영업 제안서화 (필수)
> **이유**: 현재 PPT는 데이터 나열. 사장님이 보고 "오, 큰일이네"라고 느끼지 않으면 계약 안 된다.

- 표지 슬라이드: "사장님, 지금 매달 약 N명이 경쟁사로 가고 있어요" 충격 문구 추가
- 손익분기 슬라이드 추가 (신규): 패키지별 "고객 N명만 더 오면 본전" 시각화
- 개선 전/후 예상 슬라이드 추가 (신규): 현재 점수 vs 개선 후 예상 점수
- 기존 7슬라이드 유지하되 영업 톤으로 문구 조정

**구현 위치**: `services/ppt_generator.py` 수정 (기존 파일)

### 3. 영업 우선순위 자동 분류 (필수)
> **이유**: 진단은 됐는데 "이 업체에 연락해야 하나 말아야 하나"를 리안이 하나하나 판단하면 시간 낭비다.

- 진단 완료 시 자동 태그: 1순위(핫) / 2순위(웜) / 패스
- 분류 기준 (영업_플레이북.md 기반):
  - 1순위: D등급 + (사진<10 OR 리뷰<10 OR 순위>10 OR 블로그<5) 중 2개 이상
  - 2순위: C등급 + 경쟁사 대비 절반 미만
  - 패스: A등급, 프랜차이즈 본사 직영, 최신 리뷰 1년+ 된 업체
- 이력 목록(history.html)에서 우선순위별 필터링/정렬

**구현 위치**: `services/scorer.py` 확장 + `models.py` 필드 추가 (sales_priority)

## MVP 제외 (나중에) -- 이번에 안 해도 핵심 가치 전달 가능

### 1. xlsx 배치 처리 -- 2차 업그레이드
- **제외 이유**: 리안이 아직 하루 30건도 안 보내고 있다. 단건 진단 -> 메시지 복붙 워크플로우가 먼저 매끄러워야 한다. 배치는 단건 플로우가 검증된 후에 의미 있다.
- **위험**: 배치 처리 넣으면 크롤링 부하 관리, 에러 핸들링, 큐 관리 등 엔지니어링 비용이 급격히 올라간다. MVP에서 이걸 넣으면 출시가 늦어진다.

### 2. 경쟁사 대비 상대점수 -- 2차 업그레이드
- **제외 이유**: 경쟁사 데이터를 자동으로 수집하려면 같은 지역 상위 N개 업체를 추가 크롤링해야 한다. 크롤링 시간이 5배 이상 늘어나고 네이버 차단 리스크가 커진다.
- **대안**: 1차 메시지에 "이 동네 상위 업체 평균"은 업종별 고정값(미용실 리뷰 평균 45개 등)으로 하드코딩해서 넣는다. 정확하진 않지만 영업에는 충분하다. 실제 경쟁사 비교는 리안이 방문 미팅 때 직접 보여주면 된다.

### 3. 업종별 가중치 차등 -- 2차 업그레이드
- **제외 이유**: 현재 가중치도 합리적이다. 업종별 차등은 데이터가 쌓인 후 튜닝하는 게 맞다. 지금은 계약 0건인데 가중치 최적화에 시간 쓰는 건 과잉.

### 4. 새소식 빈도/답글률/저장수 크롤링 확장 -- 2차 업그레이드
- **제외 이유**: 크롤러 셀렉터 변경 범위가 크고, 네이버 플레이스 DOM 구조가 자주 바뀌어서 유지보수 부담이 크다.

## 성공 기준 (KPI)

### 30일 목표
- **계약 5건** (월 매출 145만~445만원)
- 이를 위한 역산:
  - 1차 DM 응답률 5~10% 가정 -> 하루 30건 x 20영업일 = 600건 DM
  - 600건 x 7% 응답 = 42건 진단 결과 발송
  - 42건 x 25% 3차 진행 = 10건 패키지 제안
  - 10건 x 50% 계약 = **5건**

### 핵심 지표 (툴 관점)
| 지표 | 측정 방법 | 목표 |
|------|-----------|------|
| 단건 진단 소요 시간 | 업체명 입력 -> 메시지 복사 완료까지 | 60초 이내 |
| 1차 메시지 자동 생성 정확도 | A/B/C 버전 자동 선택이 업체 상황에 맞는 비율 | 90% |
| PPT 다운로드 -> 사장님 반응 | "어떻게 해요?" 비율 | 30% |
| 일일 DM 처리량 | 리안이 하루 동안 진단+DM 보낸 업체 수 | 30건 이상 |

### 측정 방법
- DB에 진단 건수, 생성 시각 기록 (이미 있음)
- 리안이 구글시트에 DM 발송/응답/계약 수동 기록 (영업 트래킹)

## 배포 블로커

### 1. 크롤링 안정성 미검증
- **현상**: Playwright 기반 크롤링이 네이버 DOM 변경이나 차단에 취약할 수 있다
- **해결**: 테스트 업체 10개로 크롤링 성공률 확인. 90% 미만이면 셀렉터 수정 후 재테스트. 크롤링 실패 시 수동 입력(manual.html)으로 우회 가능하므로 완벽하지 않아도 배포 가능

### 2. 네이버 검색광고 API 키 만료/한도
- **현상**: 키워드 검색량 조회에 네이버 검색광고 API 필요. 키 만료되면 키워드 점수 산출 불가
- **해결**: API 실패 시 키워드 점수를 0이 아닌 "데이터 없음"으로 처리. 나머지 6개 항목으로 종합 점수 산출하는 폴백 로직 추가

### 3. 업종별 고정값(평균 리뷰수 등) 데이터 부재
- **현상**: 메시지에 "이 동네 평균 리뷰 45개"라고 넣으려면 업종별 기준값이 필요. 현재 코드에 없다
- **해결**: 경기북부 주요 업종 5개(미용실/식당/카페/학원/피부관리)에 대해 사전 조사 후 상수로 정의. 정확하지 않아도 영업에 쓸 수 있는 수준이면 충분

### 4. 손실 고객 수 추정 로직 없음
- **현상**: 1차 메시지 B버전과 PPT 표지에 "매달 N명이 경쟁사로"라는 숫자가 필요한데, 이걸 계산하는 로직이 없다
- **해결**: `월 키워드 검색량 x (1 - 순위에 따른 CTR)` 공식으로 추정. CTR 테이블은 네이버 플레이스 일반적 분포 사용 (1위 35%, 2위 17%, 3위 12%, 4~10위 4~8%, 10위+ 1~2%)

## CTO에게 요청

### 기술 확인 필요 사항

1. **크롤링 현재 성공률**: naver_place_crawler.py의 crawl_place_detail()이 실제로 돌아가는지 테스트 업체 5개로 확인 필요. 네이버 플레이스 DOM이 바뀌었을 수 있다.

2. **메시지 생성 모듈 설계**: message_generator.py를 scorer.py와 같은 레벨에 둘지, 별도 서비스로 분리할지 아키텍처 판단.

3. **PPT 수정 범위 산정**: ppt_generator.py가 현재 약 400줄 수준. 슬라이드 2개 추가 + 기존 슬라이드 문구 변경이 기존 코드를 깨뜨리지 않는지 확인.

4. **models.py 스키마 변경 영향도**: sales_priority 컬럼 추가 + 업종별 기준값 관련 테이블 추가 시 기존 DB 마이그레이션 방법. 현재 DB_RESET=true 로 초기화하는 방식인데, 이력 데이터 보존이 필요한 시점인지 판단.

5. **손실 고객 추정 공식 검증**: 위에서 제안한 CTR 기반 추정이 기술적으로 구현 가능한지, 키워드 검색량 데이터가 있을 때/없을 때 각각 어떻게 처리할지.

## 최종 판단

**이 프로젝트의 아하 모먼트는 "메시지 자동 생성"이다.**

진단 점수나 PPT는 이미 있다. 리안이 매일 겪는 병목은 "진단은 됐는데 메시지 쓰는 게 귀찮다"이다.
업체명 하나 넣으면 1~4차 메시지가 바로 나와서 복사-붙여넣기만 하면 되는 순간,
이 툴은 "있으면 좋은 것"에서 "없으면 영업 못 하는 것"으로 바뀐다.

PPT 강화와 우선순위 분류는 이 메시지 생성과 함께 묶어서 한 번에 나가야 의미가 있다.
배치 처리는 단건 플로우가 매끄러워진 뒤에 해도 늦지 않다.

**범위를 줄여라. 메시지 + PPT + 분류, 이 3개만 이번에 확실하게 끝내라.**
