# 최종 리포트 — [네이버 플레이스 자동 진단 + PPT 제안서 생성 툴]

## 전체 평가: [조건부 준비 완료]
**사유**: 핵심 기능(크롤링 → 진단 → PPT 다운로드)은 코드로 관통되었으나, 기획 합의 단계에서 제외하기로 한 기능들이 설계/코드에 잔존해 있어 아키텍처 불일치가 존재합니다. 또한, 브라우저 자동화(Playwright)의 특성상 동시 접속에 대한 서버 다운 리스크가 방치되어 있어 배포 전 안전장치 마련이 필수적입니다.

## 발견된 엣지케이스 + 보완 필요 (최소 5개)

| 분류 | 엣지케이스 상황 | 예상되는 문제 | 보완책 |
|---|---|---|---|
| **CRITICAL** | **다수 영업사원의 동시 크롤링 요청** | Playwright는 메모리를 많이 씁니다. 5명이 동시에 검색하면 서버 메모리 초과(OOM)로 즉시 다운됩니다. (세마포어 미적용 상태) | `/crawl/start` 진입 시 `asyncio.Semaphore(3)` 등을 통해 동시 작업 수를 3~5개로 제한해야 함. |
| **CRITICAL** | **네이버 플레이스 A/B 테스트 및 UI 변경** | 정규식(Regex)에 의존하므로, 네이버가 "방문자리뷰"를 "방문자 리뷰"로 띄어쓰기만 바꿔도 크롤링이 100% 실패합니다. | 정규식 파싱 시 여백, 태그 변경에 유연하게 대응하도록 정규식 완화 및 에러율 모니터링 도입. |
| **HIGH** | **동일 상호명 검색 (예: "스타벅스")** | 검색 결과가 수십 개일 경우, `naver_place_crawler.py`가 올바른 3개를 어떻게 추려내는지 기준이 모호합니다. | 검색어에 "지역명+상호명"을 유도하는 플레이스홀더 강화 및 검색 결과 페이징/필터링 조건 추가 필요. |
| **HIGH** | **PPT 생성 중 문자열 길이 초과** | 개선 포인트(`improvement_points`) 텍스트가 비정상적으로 길 경우, PPT 도형 밖으로 글자가 삐져나가거나 렌더링 에러가 발생합니다. | 텍스트 저장 전 최대 글자 수(예: 100자) 자르기(Truncate) 로직 추가. |
| **MEDIUM** | **신규 오픈 업체 (리뷰 0개)** | 리뷰가 0개인 신규 업체는 즉시 점수 D등급을 받게 되어, 영업 시 "우리 가게는 신규라 당연한데?"라는 반발을 살 수 있습니다. | "데이터 부족(신규 업체)" 상태값을 추가하여 D등급 대신 "진단 불가/신규전략 필요"로 분기 처리. |

## 보안 최종 확인

| 확인 항목 | 상태 | 설명 |
|---|---|---|
| **Path Traversal 방지** | ✅ 양호 | PPT 다운로드 시 `os.path.basename`을 사용하여 임의 폴더 접근 방지됨. |
| **네이버 차단(IP Block) 방어** | ⚠️ 경고 | User-Agent 로테이션은 있으나, 고정 IP에서 지속 호출 시 네이버에 차단당할 확률이 매우 높음. |
| **Rate Limiting (도스 공격 방어)** | ❌ 취약 | 악의적이거나 조급한 내부 직원이 버튼을 연타할 경우 제한하는 방어 로직 없음. |
| **DB SQL Injection** | ✅ 양호 | SQLAlchemy ORM을 사용하여 기본적인 SQL 인젝션 방어됨. |

### [오늘: 즉시 지시할 사항]
1. **개발팀에 데드코드 삭제 지시**: "기획에서 빼기로 한 '검색광고 API'랑 '이력 조회 화면'이 설계서(CLAUDE.md)와 DB 모델에 남아있습니다. 나중에 꼬이지 않게 지금 당장 코드에서 지워주세요."
2. **동시 접속 제한 추가 지시**: "직원 3명이 동시에 진단 돌리면 서버 뻗는다고 합니다. 한 번에 최대 3개까지만 돌아가게 제한(Semaphore) 걸어주세요."

### [이번 주: 배포 전 세팅]
3. **서버 스펙 확인**: 이 프로그램은 램(RAM)을 많이 먹습니다. 배포할 사내 서버나 클라우드 서버의 램이 4GB 이상인지 확인하세요.
4. **한글 폰트 설치**: 서버에 '맑은 고딕'이나 '나눔 고딕' 폰트가 안 깔려 있으면 다운로드된 PPT의 한글이 다 깨집니다. 개발자에게 서버에 폰트 설치했는지 물어보세요.
5. **PPT 자동 삭제 설정**: "만들어진 PPT 파일들이 서버에 계속 쌓이면 용량 꽉 차서 멈춥니다. 매일 새벽에 24시간 지난 PPT 지워주는 세팅(Cron) 추가해주세요."

### [출시 전: 영업팀 안내]
6. **디자인 기대치 조절**: "디자이너가 만든 화려한 PPT가 아니라, 프로그램이 자동으로 네이버 컬러(파랑/초록)에 맞춰 깔끔하게 뽑아주는 형태입니다. 예쁜 디자인 템플릿 적용은 다음 버전(v1.1)에 들어갑니다."라고 영업팀에 미리 공지하세요.

## 독립 검증자 종합 의견

솔직하게 말씀드리면, **MVP(최소 기능 제품)로서는 매우 훌륭하게 뼈대가 잡혔습니다.** AI(Claude)가 기획부터 QA까지 진행하면서 발생할 수 있는 치명적 버그(DB 충돌, 템플릿 에러 등)를 QA 단계에서 스스로 꽤 잘 잡아냈습니다. 현재 상태로 서버에 올리면 **"3분 안에 PPT가 떨어지는 매직"**은 확실히 경험할 수 있습니다.

**하지만 나쁜 점(위험 요소)도 명확합니다.**
합의 과정에서 "빼자"고 했던 기능들이 코드에 찌꺼기처럼 남아있고(데드코드), 브라우저를 띄워 크롤링하는 무거운 작업을 제어하는 '안전장치(동시접속 제한, 파일 자동삭제)'가 빠져있습니다. 이대로 주면 직원들이 신기해서 하루에 수십 번씩 돌려볼 텐데, 반나절 만에 서버가 다운될 겁니다.

**가장 먼저 해결할 3가지:**
1. **서버 보호 로직 추가 (동시 크롤링 3개 제한, PPT 24시간 후 자동 삭제)**
2. **CLAUDE.md 스키마 정리 (안 쓰는 키워드 API, 이력 UI 관련 코드 덜어내기)**
3. **서버 한글 폰트 설치 및 RAM 4GB 확보**

위 세 가지만 CTO에게 전달하여 조치하시면, 다음 주 영업팀 회의에 훌륭한 무기로 공개하실 수 있습니다.
