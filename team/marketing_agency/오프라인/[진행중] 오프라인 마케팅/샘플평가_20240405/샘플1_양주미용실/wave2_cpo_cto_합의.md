# Wave 2 — CPO x CTO 크로스 토론 합의문

**날짜**: 2026-03-26
**참여**: CPO (제품 전략) / CTO (기술 설계)

---

## 토론 과정

### 쟁점 1: 경쟁사 비교 크롤링 — MVP 포함 여부

**CPO 주장 (제외):**
> 경쟁사 데이터를 자동 수집하면 크롤링 시간 5배, 네이버 차단 리스크 증가. MVP에서는 업종별 고정값(미용실 리뷰 평균 45개 등)을 하드코딩하면 영업에 충분하다. 경쟁사 비교는 "있으면 좋은 것"이지 "없으면 못 파는 것"이 아니다.

**CTO 반론:**
> CPO가 우려하는 "크롤링 5배 증가"는 상세 페이지를 5개 추가 크롤링할 때 이야기다. 내 설계는 검색 결과 페이지 1회만 조회한다. 기존 크롤링 플로우에 검색 결과 파싱 1회만 추가되므로, 추가 시간은 3~5초 수준이다. 차단 리스크도 검색 1회 추가에 불과하므로 무시 가능하다.

**CPO 재반론:**
> CTO의 "검색 페이지만" 방식이라면 리스크가 낮다는 건 인정한다. 하지만 검색 결과 요약 데이터만으로 의미 있는 비교가 가능한가? 사진수는 검색 결과에 안 나올 수 있고, 편의기능도 안 나온다. 결국 리뷰수 정도만 비교 가능할 텐데 그게 슬라이드 1장 추가할 만큼 가치가 있나?

**CTO 최종 응답:**
> 검색 결과에서 확실히 추출 가능한 것: 리뷰수, 블로그수, 순위(노출 위치 자체가 순위). 사진수와 편의기능은 못 가져온다. 하지만 영업 제안서에서 "이 동네 1위 업체는 리뷰 120개인데 사장님은 8개"라는 한 문장이면 충분하다. 하드코딩 평균값보다 실제 데이터가 훨씬 설득력 있다.

**합의:**
경쟁사 비교를 MVP에 **포함하되, CTO의 경량 방식(검색 페이지 1회 파싱)으로 한정**한다. 상세 페이지 진입은 절대 하지 않는다. 수집 항목은 리뷰수, 블로그수, 노출 순위 3개로 제한한다. 사진수/편의기능 비교는 2차에서 한다.

### 쟁점 2: xlsx 배치 처리 — MVP 포함 여부

**CPO 주장 (제외):**
> 단건 플로우가 매끄러워야 배치가 의미 있다. 리안이 아직 하루 30건도 안 보내고 있다. 배치 넣으면 큐 관리, 에러 핸들링 등 엔지니어링 비용이 급증한다. MVP 출시가 늦어진다.

**CTO 반론:**
> CPO의 논리가 틀린 건 아니지만, 현실을 안 보고 있다. 리안이 이미 2,700건 엑셀을 보유하고 있다. 영업 플레이북에도 "월요일에 50건 진단"이 루틴으로 잡혀 있다. 단건으로 50건을 하나하나 입력하면 그것만 3시간이다. 배치 없으면 실제 영업 루틴 자체가 돌아가지 않는다.

**CPO 재검토:**
> 2,700건 보유 + 주간 50건 진단 루틴... 이건 맞다. 단건만 있으면 월요일에 50건 진단하는 게 현실적으로 불가능하다. 배치 없이 "하루 30건 DM"이라는 KPI 자체가 허공의 숫자가 된다.

> 하지만 전체 배치 시스템(업로드 UI + 진행률 폴링 + 결과 목록 페이지)은 여전히 무겁다. 타협점이 필요하다.

**CTO 타협안:**
> 최소 배치로 가자. 화려한 업로드 UI, 진행률 바, 별도 결과 페이지는 다 빼자. MVP에서는:
> 1. xlsx 파일 경로를 config 또는 CLI 인자로 지정
> 2. 파싱 후 순차 크롤링 + 진단 (3초 딜레이 유지)
> 3. 결과는 기존 history 목록에 합쳐서 표시
> 4. 영업 우선순위 태그 자동 부여
>
> UI 업로드가 없으니 CDO/FE 작업이 0이다. 순수 백엔드 작업만으로 배치가 돌아간다.

**CPO 최종 판단:**
> 이 방식이면 수용한다. "리안이 파일 경로만 넣으면 알아서 돌아감"이면 충분하다. 드래그앤드롭 UI는 2차에서 만들자.

**합의:**
xlsx 배치를 MVP에 **포함하되, 백엔드 전용 경량 버전**으로 한다.
- UI 업로드 페이지 없음 (2차)
- 진행률 폴링 UI 없음 (2차)
- API 엔드포인트 1개(`POST /batch/start`에 파일 경로 전달) + 상태 조회 1개
- 결과는 기존 history 목록에 통합
- 1회 최대 50건 제한, 업체 간 3초 딜레이

### 쟁점 3: 업종별 가중치 차등 — MVP 포함 여부

**CPO 주장 (제외):**
> 계약 0건인데 가중치 최적화에 시간 쓰는 건 과잉이다. 현재 가중치도 합리적이다. 데이터가 쌓인 후 튜닝하는 게 맞다.

**CTO 반론:**
> 이건 "최적화"가 아니라 "기본 상식"이다. 미용실은 사진이 생명이고, 식당은 리뷰가 생명이다. 영업_플레이북에도 "업종별 추천 우선순위"가 이미 정리되어 있다. industry_weights.py를 딕셔너리 하나 만드는 거라 구현 비용이 거의 0이다. 30분이면 끝난다.

**CPO 재검토:**
> 구현 비용이 30분이라면 제외할 이유가 없다. 다만 조건이 있다:
> 1. 업종 판별이 불확실할 때는 반드시 default 가중치를 사용한다 (잘못된 업종 매핑으로 엉뚱한 점수가 나오면 신뢰도 타격)
> 2. 가중치 자체는 CTO가 제안한 값을 초기값으로 쓰되, 리안이 1개월 운영 후 피드백으로 조정한다

**합의:**
업종별 가중치 차등을 MVP에 **포함한다**.
- `config/industry_weights.py`에 딕셔너리로 관리
- 업종 판별 실패 시 default 가중치 사용 (절대 에러 발생 안 시킴)
- 업종 매핑은 fuzzy 매칭 (부분 문자열 포함)
- 초기값은 CTO 제안 그대로, 1개월 후 리안 피드백으로 튜닝

### 쟁점 4: 영업 메시지 생성기 아키텍처

**CPO 질문:**
> 서비스 파일 분리(message_generator.py + message.py) vs 기존 라우터 확장, 어느 게 맞나?

**CTO 주장 (분리):**
> 분리가 맞다. 이유:
> 1. 메시지 생성은 진단/크롤링과 독립적 관심사다
> 2. 진단 1회 후 메시지는 여러 번 재생성할 수 있어야 한다 (톤 조정, 데이터 변경 등)
> 3. crawl.py에 넣으면 크롤링 라우터가 500줄 넘어간다. 유지보수 지옥이 된다

**CPO 동의:**
> 납득한다. 다만 라우터가 너무 많아지면 리안이 혼란스러울 수 있다. 화면에서는 "진단 결과 페이지 안에 메시지 탭"으로 보여야 한다. 별도 페이지로 분리하지 마라.

**합의:**
- 백엔드: `services/message_generator.py` + `routers/message.py`로 **분리**
- 프론트엔드: 별도 페이지 없이 **진단 결과 페이지 내 탭**으로 통합 표시
- 메시지 캐싱: DiagnosisHistory.messages JSON 컬럼에 저장, regenerate 파라미터로 재생성

### 쟁점 5: 경쟁사 비교 크롤링 실패 시 폴백

**CPO 추가 제기:**
> 경쟁사 크롤링이 실패하면(네이버 차단, DOM 변경 등) 전체 진단이 실패하면 안 된다. 경쟁사 데이터 없이도 진단 + 메시지 + PPT가 나와야 한다.

**CTO 동의:**
> 당연하다. 경쟁사 크롤링은 Optional 단계로 구현한다. 실패 시:
> - 경쟁사 데이터: CPO가 원래 제안한 업종별 고정 평균값으로 폴백
> - PPT 경쟁사 비교 슬라이드: "업종 평균 기준" 표기
> - 1차 메시지 A버전(리뷰 격차): 고정 평균값 기준으로 판별
> - 진단 자체는 정상 완료

**합의:**
경쟁사 크롤링 실패 시 **업종별 고정 평균값으로 자동 폴백**. 진단 플로우가 끊기면 안 된다. 고정 평균값은 industry_weights.py에 함께 정의한다.

## 경쟁사 비교 구현 방식 확정

1. 타겟 업체의 address + category로 검색 쿼리 생성
2. 네이버 플레이스 검색 결과 페이지 **1회만** 조회
3. 상위 5개 업체의 **리뷰수, 블로그수, 노출 순위**만 추출
4. 상세 페이지 진입 **절대 금지**
5. 크롤링 실패 시 `config/industry_weights.py`의 **고정 평균값으로 폴백**
6. PPT 경쟁사 비교 슬라이드: 실제 데이터면 "지역 상위 5개 기준", 폴백이면 "업종 평균 기준" 표기

## xlsx 배치 구현 방식 확정

1. `POST /batch/start` -- body에 파일 경로 전달 (UI 업로드 아님)
2. openpyxl로 파싱, 업체명 컬럼 자동 감지 ("업체명", "상호", "가게명" 등)
3. 순차 크롤링 + 진단, 업체 간 3초 딜레이
4. 1회 최대 50건
5. 개별 실패 시 skip, 나머지 계속 진행
6. 결과는 기존 DiagnosisHistory에 저장, history 목록에서 통합 조회
7. `GET /batch/status/{batch_id}` -- 진행 상태 조회
8. 완료 후 영업 우선순위 태그 자동 부여

## 영업 메시지 생성기 확정

### 아키텍처
- `services/message_generator.py` -- SalesMessageGenerator 클래스
- `routers/message.py` -- `GET /message/{history_id}`, `POST /message/regenerate`
- 프론트엔드: 진단 결과 페이지 내 "영업 메시지" 탭 (별도 페이지 아님)

### 1차 메시지 자동 버전 선택 로직
- A (리뷰 격차형): 경쟁사 평균 리뷰 >= 타겟 리뷰 x 3 (CTO 원안 x5에서 완화 -- 현실적 기준)
- B (손실 고객형): 순위 > 10 AND 월검색 >= 500
- C (방치형): 기타 모든 경우 (기본값)
- 경쟁사 데이터 없으면 고정 평균값으로 A 판별, 판별 불가 시 C

### 메시지 캐싱
- 첫 생성 시 DiagnosisHistory.messages에 JSON 저장
- 재요청 시 캐시 반환
- `regenerate=true` 파라미터로 강제 재생성

## Engineering Rules 추가/수정

CTO 원안의 8개 규칙에 아래 3개 추가:

9. **경쟁사 크롤링은 Optional**: 실패해도 진단 플로우가 끊기면 안 된다. 고정 평균값 폴백 필수.
10. **배치는 API 전용**: 이번 MVP에서 배치 UI 페이지를 만들지 않는다. `/batch/start` 엔드포인트만.
11. **업종 판별 실패 시 default**: 매핑 안 되는 업종은 조용히 default 가중치 적용. 에러 발생 금지.

## CPO/CTO 서명

**CPO**: 범위를 3개에서 6개로 늘렸지만, 경쟁사 비교와 배치 모두 경량 버전으로 한정했으므로 수용한다. 핵심은 여전히 "메시지 자동 생성"이다. 이것만 제대로 되면 나머지는 부가 가치다. 만약 일정이 밀리면 4번(경쟁사), 5번(배치) 순서로 드랍한다.

**CTO**: CPO의 "드랍 우선순위" 동의한다. 경쟁사 비교와 배치는 독립 모듈이라 나중에 붙이기 쉽다. 메시지 생성 + PPT 강화 + 우선순위 분류가 코어이고, 경쟁사/배치는 부착형이다. 일정 압박 시 코어 3개 먼저 완성하고, 경쟁사/배치를 이어 붙인다.

**드랍 우선순위** (일정 밀릴 때):
1. 절대 빠지면 안 됨: 영업 메시지 자동 생성
2. 절대 빠지면 안 됨: PPT 영업 제안서 강화
3. 절대 빠지면 안 됨: 영업 우선순위 자동 분류
4. 일정 밀리면 첫 번째로 드랍: 경쟁사 비교 (고정 평균값 폴백으로 대체)
5. 일정 밀리면 두 번째로 드랍: xlsx 배치 (단건으로 우선 운영)
6. 업종별 가중치 차등: 구현 비용 15분이라 드랍 대상 아님
