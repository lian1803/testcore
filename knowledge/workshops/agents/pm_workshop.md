### 완벽한 User Story 작성법

#### INVEST 원칙 — 좋은 User Story의 6가지 기준

Bill Wake가 정립한 INVEST는 User Story 품질을 판단하는 체크리스트다. **6개 중 하나라도 빠지면 스토리가 아니다.**

| 기준 | 의미 | 실패 예시 | 수정 방향 |
|------|------|-----------|-----------|
| **I**ndependent | 다른 스토리에 의존하지 않음 | "로그인 후 대시보드 보기" (로그인 스토리에 종속) | 로그인 완료 상태를 전제(Given)로 분리 |
| **N**egotiable | 구현 방법은 팀이 협의 가능 | "반드시 모달로 구현할 것" | "사용자가 상세 정보를 볼 수 있다" (방법은 열어둠) |
| **V**aluable | 사용자 또는 비즈니스에 실제 가치 | "DB 스키마 마이그레이션" (기술 태스크) | 가치가 없으면 Epic/Task로 분류, Story 아님 |
| **E**stimable | 개발 공수를 추정할 수 있음 | "AI가 최적 경로를 찾는다" (범위 불명확) | 알고리즘 범위, 데이터 크기 명시 |
| **S**mall | 1 Sprint 안에 완료 가능 | "사용자 관리 전체 구현" | 회원가입 / 로그인 / 프로필 수정으로 분리 |
| **T**estable | 완료 여부를 명확히 판단 가능 | "사용자 경험을 향상시킨다" | "3초 내 로딩 완료, 에러 시 메시지 표시" |

#### User Story 작성 공식

```
As a [구체적인 사용자 유형],
I want to [명확한 행동/기능],
So that [얻고자 하는 가치/이유].
```

**나쁜 예:**
```
As a user, I want a better dashboard.
→ 누가? 뭘? 왜? 전부 불명확. 이건 Story가 아니라 아이디어다.
```

**좋은 예 (SaaS 마케팅 도구):**
```
As a 인하우스 마케터,
I want to see my campaign's CTR trend for the last 30 days on a single chart,
So that I can quickly identify underperforming campaigns without exporting data.
```

#### Given-When-Then (GWT) — Acceptance Criteria 작성 공식

BDD(Behavior-Driven Development)에서 온 형식. 개발자·QA·PM이 동일한 기준으로 "완료"를 판단한다.

```
Given [초기 상태/전제 조건]
When  [사용자가 취하는 행동]
Then  [시스템이 보여줘야 할 결과]
And   [추가 결과 — 필요 시]
```

**실제 예시 — 캠페인 리포트 필터 기능:**

```gherkin
# 정상 케이스 (Happy Path)
Given 마케터가 대시보드에 로그인되어 있고
      캠페인 데이터가 1개 이상 존재할 때
When  날짜 범위를 "지난 30일"로 설정하고 [적용] 버튼을 클릭하면
Then  해당 기간의 캠페인 목록과 CTR 차트가 갱신된다
And   URL 파라미터에 ?range=30d 가 반영된다

# 엣지 케이스 — 데이터 없음
Given 선택한 기간에 캠페인이 없을 때
When  [적용] 버튼을 클릭하면
Then  "해당 기간에 캠페인이 없습니다" 메시지와 새 캠페인 만들기 CTA가 표시된다

# 엣지 케이스 — 네트워크 오류
Given 필터를 적용하는 중 네트워크가 끊기면
When  API 응답이 timeout되면 (5초 초과)
Then  "데이터를 불러올 수 없습니다. 다시 시도해주세요" 토스트 메시지가 표시된다
And   이전 데이터는 유지된다 (빈 화면으로 바뀌지 않음)
```

**완성된 User Story 한 장 (실전 템플릿):**

```markdown
## US-042: 캠페인 날짜 필터

**Story:** 인하우스 마케터로서, 날짜 범위를 필터링해서 캠페인 성과를 보고 싶다.
그래야 특정 기간의 ROI를 빠르게 파악할 수 있기 때문이다.

**우선순위:** Must
**예상 공수:** 3 story points

**Acceptance Criteria:**
- [ ] Given 로그인 상태, When 날짜 범위 선택 후 적용, Then 차트와 목록 갱신
- [ ] Given 데이터 없는 기간, Then Empty State + CTA 표시
- [ ] Given 네트워크 오류, Then 에러 토스트 + 기존 데이터 유지
- [ ] 필터 상태가 URL에 반영되어 공유/북마크 가능

**Out of Scope:** 커스텀 날짜 입력 (별도 스토리 US-043)
```

### 화면 플로우 설계 원칙

#### Happy Path + Edge Cases 빠짐없이 정의하는 체계

10년차 PM의 핵심 습관: **"Happy Path는 누구나 그린다. 엣지 케이스를 그리는 사람이 시니어다."**

**화면 상태 5가지를 항상 정의하라:**

| 상태 | 설명 | 자주 놓치는 것 |
|------|------|--------------|
| **Loading** | 데이터 불러오는 중 | 스켈레톤 UI vs. 스피너 — 언제 어떤 걸? |
| **Empty State** | 데이터가 0개 | 그냥 빈 화면이면 안 됨. 안내 + CTA 필수 |
| **Populated** | 정상 데이터 있는 상태 | Happy Path. 모두가 여기만 그림 |
| **Error** | API 실패, 권한 없음 등 | 에러 종류별로 다른 메시지 필요 |
| **Partial** | 일부 데이터만 있는 상태 | 예: 온보딩 50% 완료, 일부 항목만 로드됨 |

**화면 플로우 설계 실전 체크리스트:**

```
화면명: 캠페인 대시보드

[ ] Happy Path: 캠페인 3개 이상 — 목록 + 차트 정상 표시
[ ] Empty State: 캠페인 0개 — "첫 캠페인을 만들어보세요" + [만들기] 버튼
[ ] Loading: 진입 시 스켈레톤 카드 3개 표시 (실제 레이아웃과 동일한 형태)
[ ] Error-API: "데이터를 불러오지 못했어요" + [재시도] 버튼
[ ] Error-Permission: 다른 워크스페이스 캠페인 접근 시 → 403 처리 (리다이렉트? 메시지?)
[ ] Partial: 일부 캠페인 데이터 미수집 상태 → 회색 처리 + "집계 중" 배지
[ ] Long content: 캠페인명 50자 초과 시 말줄임 처리 (tooltip으로 전체 표시)
[ ] Mobile: 모바일 뷰에서 차트 축소 or 스크롤 — 결정 필요
```

**User Journey Map — 핵심 흐름 설계:**

사용자가 목표를 달성하기까지의 전체 여정을 단계별로 정의한다.
현재 상태(Current State)와 이상적 상태(Future State) 두 버전을 만들어라.

```
[사용자 목표]: 이번 달 캠페인 성과를 30초 안에 파악하고 싶다

단계 1: 대시보드 진입
  - 행동: 앱 열기 → 대시보드 탭 클릭
  - 감정: "빨리 보고 싶다"
  - 현재 Pain: 로딩 8초, 로그인 재인증 필요
  - 개선 기회: 세션 유지 + 스켈레톤 UI로 체감 속도 개선

단계 2: 기간 필터 설정
  - 행동: 날짜 범위 선택
  - 감정: "이게 맞나?" (기본값이 왜 지난 7일?)
  - 현재 Pain: 기본값이 7일, 매번 30일로 바꿔야 함
  - 개선 기회: 마지막 사용 필터 기억 or 사용자별 기본값 설정

단계 3: 결과 확인
  - 행동: CTR 낮은 캠페인 발견
  - 감정: "이걸 어떻게 고치지?"
  - 현재 Pain: 클릭 후 편집 화면으로 이동하는 버튼이 없음
  - 개선 기회: 차트에서 바로 캠페인 편집 진입 가능하게

결론: 30초 목표 달성을 막는 병목 = 로딩 시간 + 필터 기본값 + 편집 동선
```

### 태스크 분해 + 우선순위 결정

#### Epics → Stories → Tasks 분해 구조

```
Epic (대주제, 1~3개월)
  └─ Story (사용자 가치 단위, 1~2주)
       └─ Task (개발 작업 단위, 1~3일)
            └─ Sub-task (선택, 반나절 이하)
```

**실제 SaaS 예시 — "캠페인 리포트 기능" Epic:**

```
Epic: 캠페인 성과 리포팅 (예상 2개월)

Story 1: 캠페인 목록 + 기본 지표 표시 (Must / 5 SP)
  Task 1-1: BE — /api/campaigns 엔드포인트 (GET, 페이지네이션) [2일]
  Task 1-2: FE — 캠페인 목록 테이블 컴포넌트 [1일]
  Task 1-3: FE — Empty State 컴포넌트 [0.5일]
  Task 1-4: FE — 로딩 스켈레톤 [0.5일]

Story 2: 날짜 범위 필터 (Must / 3 SP)
  Task 2-1: FE — DateRangePicker 컴포넌트 [1일]
  Task 2-2: BE — 날짜 파라미터 쿼리 처리 [1일]
  Task 2-3: FE — URL 파라미터 동기화 [0.5일]

Story 3: CTR 트렌드 차트 (Should / 5 SP)
  Task 3-1: BE — 일별 집계 API [1.5일]
  Task 3-2: FE — LineChart 컴포넌트 (Recharts) [1.5일]
  Task 3-3: FE — 모바일 반응형 [1일]

Story 4: 캠페인별 상세 리포트 (Could / 8 SP)
  → 이번 스프린트에 넣지 않음. 다음 스프린트 후보.
```

#### RICE 우선순위 — 숫자로 결정한다

```
RICE Score = (Reach × Impact × Confidence) / Effort

Reach:      이 기능이 영향 미치는 사용자 수 (월 기준)
Impact:     전환/리텐션에 미치는 영향 (3=대, 2=중, 1=소, 0.5=최소)
Confidence: 추정 확신도 (100%=확실, 80%=높음, 50%=중간)
Effort:     개발 공수 (사람×월 단위)

예시:
날짜 필터:  Reach=500, Impact=2, Confidence=80%, Effort=0.5 → RICE=1,600
CSV 내보내기: Reach=200, Impact=1, Confidence=50%, Effort=1 → RICE=100
알림 기능:  Reach=500, Impact=3, Confidence=50%, Effort=3 → RICE=250

결론: 날짜 필터 먼저. RICE가 15배 높다.
```

#### MoSCoW — 빠른 우선순위 정리

| 등급 | 기준 | 이번 릴리즈 | 예시 |
|------|------|------------|------|
| **Must** | 없으면 제품이 작동 안 함 | 반드시 포함 | 로그인, 데이터 조회 |
| **Should** | 중요하지만 없어도 런칭 가능 | 가능하면 포함 | 필터, 정렬, 차트 |
| **Could** | 있으면 좋지만 작은 가치 | 여유 있을 때 | 다크모드, 애니메이션 |
| **Won't** | 이번엔 안 함 (범위 밖) | 명시적으로 제외 | AI 추천, 자동화 |

**핵심 원칙:** Won't는 "절대 안 한다"가 아니라 "이번 스프린트엔 안 한다"다. 명시적으로 리스트에 올려야 범위 크리프(Scope Creep)를 막는다.

### 개발 공수 추정 (실용적)

#### 3가지 방법 — 언제 뭘 쓰나

**1. T-shirt Sizing (XS/S/M/L/XL)**

```
언제: 초기 로드맵 단계, 스프린트 계획 전 백로그 정리
장점: 빠름. 비개발자도 참여 가능. 분석 마비 없음.
단점: velocity 계산 어려움

XS = 반나절 이하  (버튼 텍스트 수정, 색상 변경)
S  = 1일 이하     (간단한 컴포넌트 추가, API 1개)
M  = 2~3일        (신규 화면 1개, 중간 복잡도 API)
L  = 1주일         (복잡한 기능, 여러 API 연동)
XL = 2주 이상     → 반드시 Stories로 쪼개라. 이 크기로 스프린트에 넣지 마라.
```

**2. Story Points (피보나치: 1, 2, 3, 5, 8, 13)**

```
언제: 팀이 3 스프린트 이상 함께 일한 경우, velocity 추적 필요 시
장점: velocity로 미래 스프린트 예측 가능
단점: 새 팀원이 적응 어려움, 숫자에 집착하는 문화 생길 수 있음

기준점(Anchor) 설정이 핵심:
  1 SP = "로그인 버튼 라벨 변경" (가장 단순한 작업)
  3 SP = "필터 1개 추가 + API 연동"
  5 SP = "신규 화면 1개 (CRUD 포함)"
  8 SP = "복잡한 상태 관리가 필요한 기능"
  13 SP = 쪼개야 한다는 신호
```

**3. 실제 시간 (시간/일 단위)**

```
언제: 긴급 핫픽스, 고객 약속된 데드라인이 있을 때
장점: 이해관계자와 커뮤니케이션 쉬움
단점: 예측 오류가 크게 느껴짐. "왜 3시간이 아니라 하루 걸렸냐" 압박.

팁: 실제 시간 추정 시 반드시 ×1.5 버퍼를 공식으로 적용해라.
    개발자가 "2시간"이라고 하면 → 공수는 3시간으로 계획해라.
```

**실용 가이드 — 어떤 단계에서 어떤 방식을?**

```
초기 기획/로드맵 단계  →  T-shirt Sizing
스프린트 계획 직전      →  Story Points (또는 T-shirt → SP 변환)
핫픽스/긴급 요청       →  실제 시간
외부 발표 자료         →  T-shirt (숫자 논쟁 방지)
```

### Sprint 계획의 현실

#### Velocity 측정

```
Velocity = 지난 스프린트에서 완료(Done)한 Story Points 합계

계산 방법:
  Sprint 1: 22 SP
  Sprint 2: 18 SP
  Sprint 3: 25 SP
  Sprint 4: 20 SP
  평균 velocity = (22+18+25+20) / 4 = 21.25 SP

→ 다음 스프린트 계획 시 21 SP 기준으로 잡는다.
→ 최근 3~5 스프린트 rolling average가 가장 현실적이다.
```

**Done의 정의(Definition of Done)가 일치해야 velocity가 의미 있다:**
```
Done = 코드 작성 + 코드 리뷰 통과 + QA 통과 + 스테이징 배포 완료
       (개발 완료만 Done으로 치면 velocity가 부풀려진다)
```

#### Buffer 확보 — 10~15% 규칙

```
팀 velocity: 20 SP
버퍼 (15%):   3 SP
실제 계획:    17 SP

버퍼가 필요한 이유:
  - 예상치 못한 버그 수정 (전 스프린트 이월)
  - 팀원 휴가/병가
  - 스프린트 중간 긴급 요청 (Hotfix)
  - 추정 오류 (항상 발생한다)
```

#### "이번 스프린트에 뭘 넣을 수 있나" 판단 프로세스

```
1단계: 백로그 우선순위 확인 (MoSCoW / RICE 기준)
       → 이번에 Must인 것들 먼저

2단계: Velocity 기준으로 SP 합산
       → Must가 17 SP 초과? → Should를 다음 스프린트로 이동
       → Must가 10 SP? → Should에서 채워라

3단계: 의존성 확인
       → Story A가 완료되어야 Story B 시작 가능한 경우 → 같은 스프린트에 넣지 마라

4단계: 팀 역량 확인 (Capacity Planning)
       → 이번 주 누군가 3일 휴가? → SP 조정
       → 신규 팀원 온보딩 주간? → velocity 50% 적용

5단계: Sprint Goal 1줄 정의
       → "이번 스프린트 끝나면 사용자가 캠페인 성과를 필터링해서 볼 수 있다"
       → Goal이 없는 스프린트는 방향성을 잃는다
```

### 개발팀과의 소통법

#### 모호함 없이 스펙 전달하기

**나쁜 스펙 (PM이 자주 하는 실수):**
```
"캠페인 성과 대시보드 만들어주세요. 깔끔하게요."
→ 개발자 입장: 뭘? 어떤 지표? 어떤 차트? 얼마나 데이터? 누가 볼 수 있어?
→ 결과: 개발자가 가정을 채워 만듦 → PM이 "이게 아닌데요" → 재작업
```

**좋은 스펙 (개발자가 추가 질문 없이 시작할 수 있는 수준):**
```
기능명: 캠페인 대시보드 — 날짜 필터
화면: /dashboard (기존 화면에 추가)

UI:
  - 우측 상단 DateRangePicker (Preset: 오늘 / 7일 / 30일 / 직접입력)
  - 기본값: 지난 30일
  - 선택 후 자동 적용 (버튼 없이) → URL 파라미터 동기화

API:
  GET /api/campaigns?start_date=2026-03-01&end_date=2026-03-31
  Response: { campaigns: [...], total_count: 42 }

엣지 케이스:
  - 빈 결과: "해당 기간에 캠페인이 없습니다" + [새 캠페인 만들기] 버튼
  - 로딩: 스켈레톤 카드 (기존 CampaignCard 스켈레톤 컴포넌트 재사용)
  - 에러: 기존 ErrorToast 컴포넌트 사용

완료 기준:
  - [ ] 필터 적용 시 목록 갱신
  - [ ] URL 파라미터 반영 + 공유 시 동일 필터 상태 유지
  - [ ] 모바일 (375px 이상) 정상 동작
```

#### 질문 기대하기 — 질문이 없으면 위험하다

```
개발자가 질문을 안 하는 두 가지 이유:
  1. 스펙이 너무 명확해서 (베스트 케이스)
  2. 가정을 채워서 혼자 진행 중 (최악의 케이스)

좋은 PM의 습관:
  - 스프린트 시작 첫날: "이해 안 가는 부분 있으면 오늘이 질문할 타이밍이에요"
  - 스프린트 중반: "진행 중 막히는 부분 없어요? 스펙 불명확한 것?"
  - 절대 하지 말 것: "왜 이걸 물어봐요? 스펙에 썼잖아요"
```

#### 요구사항 변경 관리 — Change Request 원칙

```
스프린트 중 변경 요청이 오면:

작은 변경 (1 SP 미만):
  → 현재 스프린트에서 처리 가능. "기존 추정에 포함됩니다" 확인.

중간 변경 (1~3 SP):
  → 같은 크기의 기존 스토리와 교체. "US-042를 빼고 이걸 넣겠습니다" 협의.

큰 변경 (3 SP 초과):
  → 다음 스프린트로. "이번 스프린트 Goal을 지키기 위해 다음으로 넘깁니다."
  → 이해관계자에게 트레이드오프 명시: "이걸 넣으면 무엇이 빠집니다."

원칙: 스프린트 중 scope 추가는 scope 제거와 함께. 공짜 추가는 없다.
```

#### PM이 개발자에게 절대 하면 안 되는 말

```
"그냥 빠르게 해주세요" → 빠름의 기준이 없다. 언제까지? 무엇을 타협?
"간단한 거잖아요" → PM 눈에 간단한 것이 개발에 간단하지 않다.
"어제 이야기한 거 기억하죠?" → 모든 결정은 문서화. 구두는 없다.
"일단 만들고 나중에 스펙 맞춰요" → 재작업의 시작.

대신 이렇게:
"이 기능 완료 기준은 이렇습니다. 다른 해석이 있으면 지금 얘기해요."
"예상 공수가 달라진다면 미리 알려주세요. 우선순위 재조정할게요."
```
