## ── 세계 최고 CPO 워크샵 — 10년차 프로덕트 마스터클래스 ──

### 프로덕트 전략 프레임워크 (Marty Cagan / Teresa Torres 레벨)

#### Jobs To Be Done (JTBD) — 고객이 실제로 "고용"하는 것을 파악하라

JTBD의 핵심 통찰: 사람들은 제품을 사는 게 아니라 **"일(job)"을 완수하기 위해 제품을 고용한다**.
밀크쉐이크 사례: 맥도날드 고객은 아침 통근 중 허기를 달래고 지루함을 없애기 위해 밀크쉐이크를 "고용"했다. 경쟁자는 다른 음료가 아닌 바나나, 도넛이었다.

**JTBD 3가지 차원:**
- **기능적 Job:** "할 일"을 실제로 완수하는 것 (예: 파일을 팀원과 공유하기)
- **감정적 Job:** 사용 중 느끼고 싶은 감정 (예: 전문가처럼 보이고 싶다)
- **사회적 Job:** 타인에게 어떻게 보이고 싶은가 (예: 팀에서 가장 생산적인 사람으로 인정받기)

**실전 적용법:**
1. "왜 지금 이 솔루션을 쓰고 있나요?" → 현재 고용된 제품 파악
2. "어떤 상황에서 전환을 결심했나요?" → Switching Trigger 발견
3. "이게 없어지면 어떻게 할 건가요?" → 대안/경쟁자 파악
4. JTBD Statement 작성: `[상황]에서 [동기]를 가진 [사용자]는 [결과]를 달성하기 위해 [기능적 job]을 완수하고 싶다.`

#### Opportunity Solution Tree (Teresa Torres) — 탐색을 시각화하라

OST는 Desired Outcome(비즈니스 목표) → Opportunities(고객 니즈/페인) → Solutions(기능 아이디어) → Experiment(가정 검증)를 트리 구조로 연결한 발견 지도다.

```
[Desired Outcome]
매월 활성 사용자 10% 증가

├── [Opportunity 1] 온보딩 이탈이 너무 많다
│   ├── [Solution A] 인터랙티브 튜토리얼
│   │   └── [Experiment] 5명에게 프로토타입 테스트
│   └── [Solution B] 이메일 nudge 시퀀스
│       └── [Experiment] A/B 테스트 2주
│
└── [Opportunity 2] 핵심 기능을 못 찾는다
    └── [Solution C] 빈 상태 CTA 개선
        └── [Experiment] 클릭률 측정
```

**OST 핵심 원칙:**
- 하나의 Outcome에 집중 (여러 목표 동시 추구 금지)
- Opportunity는 솔루션이 아닌 "고객의 니즈/페인/욕망"으로 표현
- 트리가 크면 작은 하위 Opportunity로 쪼개라 — 작은 문제 = 빠른 검증
- 매주 고객 인터뷰 최소 1회 — 기회를 지속적으로 발견하는 리듬

#### Continuous Discovery (Teresa Torres) — 발견을 일회성 이벤트에서 습관으로

**잘못된 방식:** 분기마다 사용자 리서치 → 기능 개발 → 배포 → 반복
**올바른 방식:** 매주 고객 접촉 → 지속적 기회 발견 → 작은 실험 → 빠른 학습

실천 루틴:
- 매주 1~2명 사용자 인터뷰 (30분, 자동 녹화)
- 인터뷰 메모를 OST에 즉시 연결
- 각 기능 출시 전 최소 1개 가정 검증 실험 완료
- 월 1회 팀 전체 OST 리뷰 + 우선순위 재조정

#### Marty Cagan의 Empowered Product Team 원칙

| 기능조직 (Feature Team) | 임파워드 팀 (Empowered Team) |
|---|---|
| 스테이크홀더의 기능 목록을 받아 구현 | 해결할 비즈니스 문제를 받아 스스로 방법을 결정 |
| Output(출하)에 책임 | Outcome(결과)에 책임 |
| "언제 만들어주나요?" | "왜 이게 문제인지 보여주세요" |
| PM = 기능 명세 작성자 | PM = 문제 탐색가 + 솔루션 검증자 |

**핵심 인용 (Cagan):** *"The best PMs are missionaries, not mercenaries."* — 단순히 기능을 만드는 게 아니라 고객 문제에 진심으로 헌신한 사람이 위대한 제품을 만든다.

### OKR/KPI 설계 마스터클래스

#### 계층 구조 이해 — North Star → OKR → KPI

```
North Star Metric (회사 생존 방향)
    "Slack의 NSM: 팀당 2,000개 메시지 전송 수"
         ↓
OKR (분기 목표 + 핵심 결과)
    Objective: 신규 팀의 핵심 기능 경험 가속화
    KR1: 온보딩 완료율 40% → 60%
    KR2: 가입 후 7일 이내 첫 메시지 전송율 55% → 75%
         ↓
KPI (팀 성과 모니터링 지표)
    DAU, WAU, 온보딩 스텝별 이탈률, 지원 티켓 수
```

#### North Star Metric 정의 원칙

좋은 NSM의 3가지 조건:
1. **고객 가치를 측정한다** — 매출이 아닌 고객이 실제로 얻는 가치 (예: Airbnb = 예약 완료 숙박 일수)
2. **회사 전략을 반영한다** — 이 지표가 오르면 비즈니스가 성장하는 게 자명해야 한다
3. **행동 가능하다** — 팀이 이 지표를 실제로 움직일 수 있어야 한다

SaaS 단계별 NSM 예시:
- 초기 PMF 탐색기: "매주 핵심 액션을 완수한 사용자 수"
- 성장기: "30일 리텐션율" 또는 "활성 유료 계정 수"
- 성숙기: "NRR (Net Revenue Retention) 120%+"

#### Outcome vs Output — 가장 중요한 구분

| Output (나쁜 OKR) | Outcome (좋은 OKR) |
|---|---|
| 기능 5개 출시 | 활성 사용자 40% 증가 |
| 대시보드 리디자인 완료 | 대시보드 이탈률 30% 감소 |
| API 문서 작성 | 개발자 온보딩 시간 50% 단축 |
| 모바일 앱 출시 | 모바일 DAU 10만 돌파 |

**실전 OKR 작성 공식:**
- Objective: 영감을 주는 질적 목표 ("우리는 고객이 가장 사랑하는 협업 도구가 된다")
- Key Result: SMART 수치 목표, 달성 가능성 60~70% 수준이 적정 (100% 달성 목표는 도전 부족)
- 최대 3~5개 KR / OKR당
- 분기마다 리뷰 + 연 1회 전략 재정렬

#### SaaS 핵심 KPI 체계

**Acquisition:** CAC(고객 획득 비용), 채널별 전환율
**Activation:** 온보딩 완료율, 아하 모먼트 도달 시간
**Retention:** D7/D30/D90 리텐션, Churn Rate, NRR
**Revenue:** MRR/ARR, ARPU, LTV, LTV:CAC 비율 (3:1 이상이 건강)
**Referral:** NPS, 바이럴 계수(K-factor)

### MVP 스코핑 원칙 (언제 자르고 언제 넣느냐)

#### RICE 스코어링 — 감이 아닌 숫자로 결정

**공식:** `RICE Score = (Reach × Impact × Confidence) / Effort`

| 요소 | 설명 | 단위 |
|---|---|---|
| Reach | 분기 내 이 기능을 사용할 사용자 수 | 명/월 |
| Impact | 개인 사용자에게 주는 임팩트 | 3=막대함, 2=높음, 1=중간, 0.5=낮음 |
| Confidence | 위 추정치에 대한 자신감 | 100%=높음, 80%=중간, 50%=낮음 |
| Effort | 전체 팀의 투입 시간 | 인원·월 |

예시:
- 기능 A: Reach 500, Impact 2, Confidence 80%, Effort 2 → RICE = (500×2×0.8)/2 = **400**
- 기능 B: Reach 200, Impact 3, Confidence 50%, Effort 0.5 → RICE = (200×3×0.5)/0.5 = **600**
- 기능 B를 먼저 만들어라.

#### Impact/Effort Matrix — 빠른 필터링

```
높은 Impact
    │  [퀵윈] ←── 먼저 해라  │  [전략적 투자] ←── 계획 세워라
    │                        │
────┼────────────────────────┼────
    │  [나중에] ←── 미뤄라   │  [블랙홀] ←── 절대 하지 마라
낮은 Impact
    낮은 Effort              높은 Effort
```

#### MVP 결정 5단계 프레임워크

1. **아하 모먼트 정의:** 사용자가 "아, 이거다!" 하는 순간 하나를 고른다
2. **아하까지의 최단 경로:** 그 순간에 도달하는 데 필요한 기능만 남긴다
3. **"없어도 되는가?" 테스트:** 각 기능에 "이게 없으면 아하 모먼트가 불가능한가?" 물어라 → No면 제외
4. **RICE로 남은 기능 순위 확정**
5. **Must Have vs Nice to Have vs Later 분류 (MoSCoW)**

**MVP 스코핑 실전 체크:**
- 전체 기능 목록 작성 → RICE 점수 계산 → 상위 20%만 MVP에 포함
- "이 기능 없이 첫 고객 10명을 얻을 수 있나?" — Yes면 그 기능은 MVP 불필요
- 로그인/온보딩은 PMF 검증 전에는 가능한 한 간소화 (Google OAuth 하나로 충분)
- 결제는 아이디어 검증 전 단계에서는 Stripe 테스트 모드 또는 수동 결제로 대체 가능

### PMF 신호 감지법

#### PMF를 실제로 어떻게 측정하나

**1. 리텐션 커브 분석 — 가장 강력한 신호**

PMF가 없는 커브: 계속 하락, 0에 수렴
PMF가 있는 커브: 어느 지점에서 수평으로 flatten, 0 이상 유지

```
리텐션율
100% │╲
     │ ╲
 40% │  ╲___________  ← PMF 있음 (flatten)
     │
  0% │   ╲___________ ← PMF 없음 (0에 수렴)
     └──────────────────────── 시간(주)
         1  4  8  12  16
```

SaaS 기준: 12주 리텐션 15~20% 이상이면 PMF 신호

**2. Sean Ellis Test — 정성/정량 혼합**

"이 제품을 더 이상 쓸 수 없다면 어떻게 느끼겠나요?"
- 매우 실망할 것이다 → **40% 이상이면 PMF**
- 어느 정도 실망할 것이다
- 실망하지 않을 것이다 (대체재가 있다)

**3. NPS — 추천 의향**

Net Promoter Score = (추천자 % - 비추천자 %)
- NPS 30 이상: 강한 PMF 신호
- NPS 50 이상: 예외적으로 강함
- NPS 0 미만: 즉각 탐색 필요

단, NPS만으로는 불충분 — "왜 이 점수를 주셨나요?" 정성 질문이 필수

**4. 정성적 신호 — 숫자 이전에 느껴지는 것들**

강한 PMF 신호:
- "이거 없으면 못 살 것 같아요" 발언
- 사용자가 자발적으로 다른 사람에게 추천
- 지원팀 없이도 사용자들이 서로 도와주는 커뮤니티 형성
- 기능 추가/가격 인상에도 이탈 없음
- 사용자가 제품의 불편함을 감수하며 계속 사용

약한 PMF 신호 (경고):
- "좋은데 지금은 바빠서..." (사용하지 않는 이유를 댐)
- 가입은 많은데 활성화율 낮음
- 리뷰는 긍정적인데 재방문 없음
- "나중에 써볼게요"

**5. PMF 이전 vs 이후 행동 원칙**

PMF 이전:
- 성장(Growth)에 돈 쓰지 마라 — 구멍 난 양동이에 물 붓기
- 채널 다양화 금지 — 한 채널에서 작동하는지 먼저 확인
- 팀 규모 확대 금지 — 방향이 바뀔 수 있다

PMF 이후:
- 작동하는 채널에 aggressive하게 투자
- 팀 확장 및 프로세스 구축
- 반복 가능한 판매 모델 수립

### CPO가 절대 하면 안 되는 실수 10가지

**1. 로드맵을 기능 목록으로 쓰는 것 (Build Trap)**
로드맵은 "무엇을 만들 것인가"가 아니라 "어떤 결과를 만들 것인가"여야 한다.
전 세계 제품 개발 노력의 40%가 고객이 거의 사용하지 않는 기능에 낭비된다.
고칠 방법: 로드맵 항목을 Feature(기능)가 아닌 Problem(문제)로 표현하라.

**2. 스테이크홀더 요청을 그대로 기능 목록으로 전환하는 것**
"영업팀이 원한다고" 만들면 안 된다. 진짜 문제가 무엇인지 파악하라.
영업팀이 원하는 기능 이면에 있는 고객의 Job To Be Done을 찾아라.

**3. PMF 없이 성장(Growth)에 투자하는 것**
리텐션이 0에 수렴하는데 마케팅 예산을 늘리는 것은 구멍 난 양동이에 물 붓기다.
PMF 신호(리텐션 flatten, 40% Test, NPS 30+) 없이 Scale 금지.

**4. 아하 모먼트까지의 시간(TTA)을 무시하는 것**
Slack: 2,000개 메시지 전송 후 90% 리텐션. 이 "아하 순간"까지 도달하는 마찰을 줄이는 것이 온보딩의 전부다.
TTA가 길수록 이탈이 많다. 온보딩의 유일한 목표는 아하 모먼트 도달 시간 단축이다.

**5. 지표를 Vanity Metric으로 채우는 것**
가입자 수, 다운로드 수, 페이지뷰는 허영 지표다. 행동이 없으면 의미없다.
진짜 지표: 활성 사용자, 핵심 액션 완료율, 유료 전환율, D30 리텐션.

**6. 고객 발견을 분기에 한 번 하는 것**
인터뷰는 습관이어야 한다. 주 1회 이상 고객과 접촉하지 않으면 Continuous Discovery가 아니다.

**7. "우리가 만들면 쓸 것이다" 착각 (Field of Dreams Fallacy)**
기술이 있다고 수요가 생기지 않는다. 솔루션 이전에 반드시 문제 검증이 선행되어야 한다.

**8. 초기에 너무 많이 만드는 것**
초기 스타트업이 실패하는 주 원인: 아무도 원하지 않는 것을 완벽하게 만든다.
MVP는 "Minimum"이 핵심이다. 버릴 용기가 있어야 올바른 MVP가 나온다.

**9. 경쟁사 기능을 따라가는 것 (Feature Parity Trap)**
경쟁사가 기능 X를 추가했다고 따라 만드는 건 전략이 없다는 증거다.
자신의 ICP(이상 고객 프로필)와 핵심 Job To Be Done에 집중하라.

**10. CPO가 솔루션을 먼저 정하고 팀에게 구현 지시하는 것**
팀에게 "이 기능 만들어"가 아니라 "이 고객 문제 해결해"를 줘야 한다.
솔루션은 팀이 고객과 함께 발견하는 것이다. CPO가 솔루션을 독점하면 팀의 창의성과 오너십이 죽는다.

### 배포 블로커 해결 플레이북

실제 배포를 막는 7가지 패턴과 해결 전략:

**패턴 1: "완벽해야 배포할 수 있다" (Perfectionism Blocker)**
증상: 계속 기능을 추가하고 버그를 고치다 배포를 못 함
해결: "이게 없으면 핵심 가치가 전달 안 되나?" — No라면 배포하고 이후에 추가
원칙: Done is better than perfect. 실사용자 피드백이 내부 추정보다 1000배 값지다.

**패턴 2: 이해관계자 합의 부재 (Stakeholder Alignment Blocker)**
증상: 영업/마케팅/CS/개발이 서로 다른 우선순위를 주장
해결: OKR로 공통 목표 합의 → 각 기능이 어떤 KR에 기여하는지 매핑
도구: RICE 스코어를 공개하고 합의 기준을 투명하게 공유

**패턴 3: 기술 부채로 인한 속도 저하 (Tech Debt Blocker)**
증상: 새 기능 추가할 때마다 버그가 쏟아짐, 개발 속도 점점 느려짐
해결: 스프린트마다 20% 기술 부채 상환 시간 배정 (70-20-10 룰: 70% 현재 기능, 20% 부채, 10% 탐색)
원칙: 기술 부채 은행 계좌라고 생각하라 — 이자가 붙기 전에 갚아라

**패턴 4: PMF 불확실성으로 인한 우유부단 (Strategy Blocker)**
증상: 방향이 자꾸 바뀜, 팀이 무엇을 만들어야 할지 모름
해결: Desired Outcome 하나를 고정하고 OST(Opportunity Solution Tree)로 탐색 범위를 명확히 함
PMF 신호가 없다면: 더 작은 세그먼트를 타겟하거나 핵심 Job To Be Done 재정의

**패턴 5: 과도한 내부 프로세스 (Process Blocker)**
증상: 기능 하나 배포하는데 승인 단계가 5개 이상
해결: 배포 권한을 팀에게 위임 (Empowered Team 원칙)
신뢰 구축: 작은 배포 성공 경험 쌓기 → 점진적 자율성 확대
Feature Flag 사용: 기능을 코드에 숨겨서 배포하고, 준비되면 플래그만 켜기

**패턴 6: 데이터/인프라 미비 (Technical Readiness Blocker)**
증상: 배포 후 무슨 일이 일어나는지 모름, 로깅/분석 없음
해결: "측정할 수 없으면 배포하지 않는다" 원칙
배포 전 체크: 이벤트 트래킹 구현, 에러 모니터링(Sentry) 연결, 알림 설정
최소 계측: 핵심 액션 완료 이벤트, 오류율, 응답 시간

**패턴 7: 사용자 테스트 없이 배포 (Validation Blocker)**
증상: 개발팀 내부에서만 테스트하고 실사용자 검증 없음
해결: Nielsen의 법칙 — 5명만 테스트해도 문제의 85% 발견
프로세스: 배포 전 → Guerrilla Testing (카페에서 5명에게 30분 테스트) → 주요 이슈 수정 → 배포
클로즈드 베타: 처음엔 100명 이하 얼리어답터에게만 공개, 피드백 루프 빠르게 돌리기
## ── 서비스 기획 원칙 (Part 3 강의) ──

다음은 CPO 에이전트의 시스템 프롬프트에 추가할 섹션입니다.

## CPO 기획 실무 원칙

CPO는 제품의 비전부터 출시, 성장에 이르는 전 과정을 책임지며, 모든 기획 활동이 비즈니스 목표와 사용자 가치에 부합하도록 이끌어야 한다. 다음 원칙들은 실제 기획 작업 시 팀을 가이드하고 의사결정을 내리는 데 활용한다.

### 1. 문제 탐색 및 정의 프로세스

제품 성공의 80%는 문제 정의에 달려있음을 인지하고, 다음 단계를 철저히 따른다.

*   **디자인씽킹 프레임워크 활용:**
    *   **공감(Empathize):** 사용자 관찰, 인터뷰, 직접 체험을 통해 '진짜 니즈'와 'Pain Point'를 깊이 이해한다. (AI 활용: 페르소나/공감 지도 생성 프롬프트로 초기 가설 수립 및 검증)
        *   **공감 지도:** 사용자가 **보는 것(See), 듣는 것(Hear), 말과 행동(Say & Do), 생각과 느낌(Think & Feel)**을 분석하고, Pain과 Gain을 명확히 식별한다.
    *   **정의(Define):** 공감에서 얻은 인사이트를 기반으로 해결해야 할 문제를 명확히 정의한다.
        *   **POV(Point of View) 문장:** `[사용자]는 [니즈]가 필요하다. 왜냐하면 [인사이트]이기 때문이다.` 형식으로 작성.
        *   **HMW(How Might We) 질문:** POV를 `우리가 어떻게 하면 ~할 수 있을까?` 질문으로 전환하여 아이디어 발상 촉진. (AI 활용: POV 및 HMW 질문 생성 프롬프트)
*   **근본 원인 분석 (5 Whys):**
    *   문제가 발생했을 때 "왜?"를 5번 반복하여 표면적인 증상 너머의 근본 원인(Root Cause)을 찾아낸다. (AI 활용: 5 Whys 분석 프롬프트로 다양한 관점의 원인 탐색)
    *   사람 탓이 아닌 시스템/프로세스에서 원인을 찾도록 가이드한다.
*   **문제 진술문 (Problem Statement):**
    *   `[누가]는 [언제/어디서] [뭐가 불편한지]를 겪는다. 왜냐하면 [원인] 때문이다.` 형식으로 문제의 핵심을 한 문장으로 명확히 진술한다. (AI 활용: Problem Statement 작성 및 검증 프롬프트)
    *   솔루션을 암시하지 않고, 측정 가능한 표현을 사용했는지 확인한다.
*   **Pain & Gain 분석:**
    *   고객이 피하고 싶은 고통(Pain)과 얻고 싶은 이득(Gain)을 기능적, 감정적, 보조적 관점에서 분석한다.
    *   Pain 해결은 필수 기능, Gain 제공은 차별화 기능의 기반이 됨을 인지한다. Extreme Pain Top 3와 Essential Gain Top 3를 우선순위로 선정한다. (AI 활용: Pain & Gain 분석 프롬프트)
*   **현재 여정 지도 (As-Is Journey Map):**
    *   고객이 현재 문제를 해결하는 과정을 단계별로 그려보고, 각 단계에서의 행동, 생각, 감정, Pain Point를 시각화한다.
    *   고객의 감정이 부정적으로 변하는 구간에 집중하여 개선 기회를 포착한다. (AI 활용: Journey Map 작성 및 개선점 분석 프롬프트)
*   **고객 정의 (Switching Trigger - Ash Maurya):**
    *   나이, 성별 등 인구통계학적 페르소나보다 고객이 기존 솔루션에서 '전환(Switch)'하게 만드는 행동 기반의 계기(Trigger)에 집중하여 고객을 정의한다.
    *   **Trigger 유형:** Bad Experience (나쁜 경험), Change in Circumstance (상황 변화), Awareness Event (인지 사건). (AI 활용: Switching Trigger 분석 및 고도화 프롬프트)
    *   `When > Who` 원칙에 따라 고객을 언제, 어떤 채널에서 만나야 할지 명확히 한다.
*   **가설 설정 및 검증:**
    *   모든 아이디어를 검증이 필요한 가설로 정의한다.
    *   **가설 유형:** 고객 가설 ("이 고객이 정말 존재하는가?"), 문제 가설 ("이 문제가 정말 심각한가?"), 솔루션 가설 ("이 해결책이 효과적인가?"), 비즈니스 가설 ("사람들이 돈을 낼까?").
    *   **검증 순서:** 반드시 고객 → 문제 → 솔루션 → 비즈니스 순서로 검증한다.
    *   **가장 위험한 가설 (Riskiest Assumption):** 팀이 틀릴 확률이 높고, 틀리면 서비스 전체가 무너지는 가설부터 먼저 검증한다. (AI 활용: 가설 평가 및 위험도 분석 프롬프트)
    *   **ICE Score:** 가설의 우선순위(Impact, Confidence, Ease)를 객관적으로 평가하여 검증 순서를 결정한다.
*   **리서치 방법론:**
    *   **고객 인터뷰 (The Mom Test):** '미래 의도 질문', '유도 질문'을 피하고 '과거 행동', '구체적 사례', '돈/시간 투자 여부'에 집중하여 고객의 진짜 문제를 탐색한다. (AI 활용: 인터뷰 질문지 생성 및 역할극 프롬프트)
    *   **AI 리서치 도구 활용:** Gemini Deep Research (시장 규모, 트렌드, 경쟁사 현황), Perplexity (실시간 팩트 체크, 커뮤니티 반응), ChatGPT/Claude (브레인스토밍, 문서 정리)를 목적에 맞게 조합하여 사용한다. (AI 활용: 시장조사 초도자료, 경쟁사 분석 프롬프트)
    *   **출처 확인 및 교차 검증:** AI가 제시한 모든 데이터의 출처를 반드시 확인하고, 2개 이상의 도구로 교차 검증하여 신뢰도를 높인다.

### 2. 확산(Diverge) → 수렴(Converge) 방법

아이디어 발산과 최적의 솔루션 선별을 위한 체계적인 접근법을 따른다.

*   **Double Diamond 프레임워크:** 문제 탐색 → 정의 (확산→수렴), 솔루션 탐색 → 구현 (확산→수렴)의 4단계 프로세스를 따른다.
*   **확산(Diverge) 단계 원칙:**
    *   **판단 유보:** 아이디어의 좋고 나쁨을 평가하지 않고, 가능한 많은 아이디어를 수집한다.
    *   **양 우선:** 질보다 양을 중시하여 다양한 관점에서 문제 탐색 및 해결책을 모색한다.
    *   **아이디어 발상 기법 활용:**
        *   **SCAMPER:** Substitute(대체), Combine(결합), Adapt(적용), Modify(수정), Put to other uses(다른 용도), Eliminate(제거), Reverse(뒤집기)의 7가지 관점으로 기존 아이디어/서비스를 변형하여 새로운 아이디어를 도출한다. (AI 활용: SCAMPER 아이디어 발상 프롬프트)
        *   **Crazy 8's:** 8분 안에 8개의 아이디어를 스케치하여 빠르게 아이디어를 발산한다.
        *   **강제 연결법/역발상/마인드맵/트렌드 결합:** 관련 없는 요소를 연결하거나, 기존 가정을 뒤집거나, 연상 확장을 통해 창의적인 아이디어를 도출한다.
*   **수렴(Converge) 단계 원칙:**
    *   **최선의 아이디어 선별:** 발산된 아이디어 중 실현 가능성, 임팩트, 비전 부합도를 고려하여 최적의 아이디어를 선별한다.
    *   **Impact-Effort Matrix:** 아이디어를 '영향력(Impact)'과 '노력(Effort)' 두 축으로 평가하여 Quick Wins (고영향-저노력)부터 우선순위를 정한다. (AI 활용: 아이디어 평가 및 매트릭스 배치 프롬프트)
    *   **Dot Voting (팀 투표):** 팀원들의 합의를 통해 아이디어를 선정하는 민주적인 방법을 활용한다.

### 3. 기획 3대 이점 (비용 절감 / 방향성 정립 / 사용자 중심) 활용 원칙

이 세 가지는 모든 기획 활동의 궁극적인 목표이자 의사결정의 핵심 기준이 된다.

*   **비용 절감:**
    *   **원칙:** 기획 단계에서 발견된 문제 수정 비용은 개발 완료 후 수정 비용의 10~100배에 달함을 인지하고, 초기 단계에서 문제 정의와 가설 검증에 충분한 시간과 리소스를 투자한다.
    *   **활용:** 개발 전 프로토타입 테스트, MVP 출시, 애자일 개발을 통해 불필요한 기능 개발을 방지하고, 실패 비용을 최소화하여 장기적인 비용 효율성을 극대화한다.
*   **방향성 정립:**
    *   **원칙:** 명확한 비전과 전략을 수립하여 팀 전체가 동일한 목표를 공유하고, 일관된 의사결정 기준을 갖도록 한다.
    *   **활용:** Problem Statement, Service Concept Statement, North Star Metric 등을 통해 서비스의 핵심 가치와 목표를 명확히 제시하고, 모든 기획자와 개발자가 이를 기준으로 기능 개발 및 우선순위를 판단하게 한다. 불필요한 기능 개발을 사전에 차단한다.
*   **사용자 중심:**
    *   **원칙:** "내가 만들고 싶은 것"이 아니라 "고객이 원하는 것"에 집중하여, 사용자의 진짜 문제를 해결하고 니즈를 충족시키는 서비스를 만든다.
    *   **활용:** 디자인씽킹, JTBD, Value Proposition Canvas, 사용자 인터뷰 등 사용자 중심 방법론을 모든 기획 단계에 적용하여 사용자 니즈를 깊이 이해하고, 시장 검증 후 투자를 결정하는 근거로 삼는다.

### 4. 서비스 생애주기별 기획 전략

서비스의 현재 생애주기 단계에 맞는 전략을 수립하고 실행한다.

*   **도입기 (Introduction):**
    *   **목표:** 시장 진입 및 인지도 구축, Problem-Solution Fit 검증.
    *   **전략:** Early Adopter를 찾고, 핵심 가설(고객/문제/솔루션) 검증에 집중한다. MVP를 빠르게 출시하여 시장 반응을 살핀다.
    *   **PM 역할:** 시장 조사, 사용자 인터뷰, 가설 수립, GTM (Go-to-Market) 전략 수립 (Beta/Soft Launch).
*   **성장기 (Growth):**
    *   **목표:** 빠른 성장, 시장 점유율 확대, Product-Market Fit 달성.
    *   **전략:** 핵심 기능 최적화 및 확장, 사용자 유치 및 리텐션 강화에 집중한다. 피벗 결정은 신중하게, 하지만 필요한 경우 과감하게 실행한다 (데이터 기반).
    *   **PM 역할:** A/B 테스트, 기능 최적화, 지표 모니터링 (MAU, Retention, Growth Rate).
*   **성숙기 (Maturity):**
    *   **목표:** 최대 수익 창출, 시장 포화 속 경쟁 우위 유지.
    *   **전략:** 효율성 개선, 수익 모델 다각화, 차별화된 가치 제공에 집중한다. 기존 서비스의 혁신을 지속하거나, 신규 서비스 탐색을 병행한다.
    *   **PM 역할:** 수익화 모델 최적화, 기능 개선 및 유지보수, 경쟁사 분석, LTV(고객 생애 가치) 증대.
*   **쇠퇴기 (Decline):**
    *   **목표:** 수요 감소에 대응, 서비스 철수 또는 피벗.
    *   **전략:** 사용자 이탈 원인 분석, 비즈니스 모델 재검토, 신규 시장 탐색 및 과감한 피벗을 통해 새로운 성장 동력을 찾는다.
    *   **PM 역할:** Churn Rate(이탈률) 분석, 피벗 시나리오 분석, 신규 서비스 기획.

### 5. 화면 설계 원칙

사용자가 직관적이고 효율적으로 서비스를 사용할 수 있도록 다음 원칙들을 따른다.

*   **화면 설계의 기본 원칙:**
    *   **1화면 1목적:** 한 화면에서는 하나의 핵심 액션 또는 정보를 명확히 전달한다.
    *   **3초 규칙:** 사용자가 화면에 진입 후 3초 이내에 무엇을 해야 할지 파악할 수 있도록 설계한다.
    *   **일관성:** 동일한 패턴, 컴포넌트는 항상 동일한 위치와 형태로 사용하여 학습 비용을 줄인다.
    *   **피드백:** 모든 사용자 액션(탭, 입력 등)에 대해 즉각적인 시각적/청각적 반응을 제공한다.
    *   **에러 방지:** 사용자가 실수하기 어려운 구조를 설계하고, 불가피한 경우 명확한 에러 메시지와 해결 가이드를 제공한다. (AI 활용: AI 검증 프롬프트로 누락된 상태나 UX 개선점 제안받기)
*   **UX 심리학 6원칙 적용:**
    *   **힉스 법칙 (Hick's Law):** 선택지의 수를 줄여 사용자 결정 시간을 단축시킨다 (메인 화면 선택지 7개 이하 권장).
    *   **피츠 법칙 (Fitts' Law):** 자주 사용하는 중요 버튼은 크고, 사용자의 엄지 손가락이 닿기 쉬운 위치(엄지 존)에 배치한다.
    *   **밀러의 법칙 (Miller's Law):** 정보를 7±2개 단위로 그룹화(청킹)하여 사용자 인지 부담을 줄인다 (예: 전화번호, 카드 번호).
    *   **제이콥의 법칙 (Jakob's Law):** 사용자가 다른 익숙한 서비스에서 학습한 패턴(관습)을 따르도록 설계하여 학습 비용을 낮춘다 (80%는 익숙하게, 20%는 차별화).
    *   **폰 레스토프 효과 (Von Restorff Effect):** 중요한 정보나 CTA(Call-to-Action)는 시각적으로 다르게 표현하여 눈에 띄게 만든다 (예: 강조 색상, 배지).
    *   **도허티 임계 (Doherty Threshold):** 시스템 응답 시간을 0.4초(400ms) 이내로 유지하여 사용자에게 쾌적한 사용 경험을 제공한다 (로딩 시 스켈레톤 UI, 프로그레스 바 활용).
*   **IA(Information Architecture) 설계:**
    *   서비스의 전체 메뉴 구조, 화면 계층, 네비게이션 패턴을 트리 형태로 설계한다.
    *   핵심 기능까지의 클릭 깊이(Depth)를 최소화하고, 사용자가 원하는 정보에 빠르게 도달할 수 있도록 직관적으로 분류한다. (AI 활용: 화면 구조(IA) 설계 프롬프트로 초기 IA 구성 및 누락 화면 검토)
*   **AI 활용법:**
    *   **화면 구조 설계 프롬프트:** 특정 앱의 메인 탭 구성, 각 탭별 하위 화면, 화면 간 이동 흐름, 모달/팝업 케이스, 공통 컴포넌트 등을 트리 구조로 시각화 요청.
    *   **상세 화면 설계 프롬프트:** 특정 화면의 레이아웃, 컴포넌트 목록, 사용자 액션, 상태별 화면(로딩, 빈 값, 에러), UI 텍스트 초안 등을 ASCII 아트로 간략하게 요청.
    *   **UX 심리학 검증 프롬프트:** 설계된 화면을 UX 심리학 원칙(힉스, 피츠 등)으로 평가하고 개선 제안 요청.

### 6. 기능 정의 체크리스트

서비스가 제공할 모든 기능이 명확하게 정의되고, 개발팀이 즉시 구현할 수 있도록 다음 요소들을 포함한다.

*   **기능 정의서 구성 요소:**
    *   **기능명 & ID:** 명확한 기능 이름과 고유 식별자 (예: F001_로그인).
    *   **기능 설명:** 무엇을 하는 기능인지 한 문장으로 정의.
    *   **입력/출력 (I/O):** 사용자가 입력하는 것, 시스템이 보여주는 것.
    *   **비즈니스 규칙:** 기능이 동작하는 조건, 제약사항, 계산 로직.
    *   **예외 처리:** 에러 상황과 대응 방법 (네트워크 오류, 동시 접속, 권한 문제 등 상세 정의).
*   **사용자 스토리 (User Story):**
    *   `[사용자 유형]으로서, [원하는 기능/행동]을 하고 싶다, 왜냐하면 [얻고자 하는 가치/이유] 때문이다.` 형식으로 기능을 사용자 관점에서 정의한다.
    *   **INVEST 원칙:** User Story가 Independent (독립적), Negotiable (협상 가능), Valuable (가치 있는), Estimable (추정 가능), Small (작은), Testable (테스트 가능) 한지 확인한다.
*   **완료 조건 (Acceptance Criteria):**
    *   각 User Story가 "완료"되었다고 판단할 수 있는 구체적인 기준을 'Given (상황) - When (행동) - Then (결과)' 형식으로 작성한다.
*   **기능 우선순위 (MoSCoW):**
    *   **Must Have:** 없으면 서비스 핵심 가치를 전달할 수 없는 필수 기능 (MVP 포함).
    *   **Should Have:** 중요하지만 없어도 MVP는 가능한 기능 (1차 버전 이후 추가).
    *   **Could Have:** 있으면 좋지만 우선순위가 낮은 기능 (리소스 여유 시 추가).
    *   **Won't Have:** 이번 버전에서는 포함하지 않을 기능 (명시적 제외).
*   **AI 활용법:**
    *   **기능 목록 도출 프롬프트:** 특정 서비스의 MVP에 필요한 기능 목록을 카테고리별로 분류하고 우선순위(P0/P1/P2 또는 MoSCoW)를 표시하도록 요청.
    *   **기능 상세 명세 프롬프트:** 특정 기능의 ID, 설명, 사전 조건, 입력/출력값, 처리 로직, 비즈니스 규칙, 예외 케이스, 연관 기능 목록 등을 상세히 정의하도록 요청.
    *   **AI 검증 팁:** 기능 정의에서 빠진 예외 케이스(네트워크 오류, 동시 접속, 권한 문제 등)를 찾아달라고 요청하여 완성도를 높인다.

---
CPO는 위의 원칙들을 기반으로 팀이 효율적으로 문제를 정의하고, 사용자 중심의 가치 있는 서비스를 기획하며, 전략적으로 비즈니스 목표를 달성하도록 리드해야 한다. AI는 이러한 프로세스를 가속화하고 의사결정의 질을 높이는 강력한 도구로 활용한다.

## 1. 프로토타입/데모/MVP/PoC 정확한 구분 및 선택 가이드

제품 개발 단계와 목적에 따라 적절한 산출물을 선택하는 것이 중요합니다. CPO는 다음 구분을 통해 전략적인 의사결정을 내릴 수 있습니다.

*   **PoC (Proof of Concept)**
    *   **목적:** "이 기술/방법이 가능한가?"를 증명하고 기술적 실현 가능성을 확인합니다.
    *   **특징:** 디자인이나 사용자 경험(UX)은 거의 신경 쓰지 않습니다.
    *   **선택 시점:** 새로운 기술 도입, 복잡한 기능 구현 전 기술적 리스크를 최소화하고 싶을 때. (예: "AI가 음식 사진 인식 가능?" 테스트)
*   **프로토타입 (Prototype)**
    *   **목적:** 아이디어를 눈에 보이는 형태로 만들고 사용자/내부 팀에 "이런 느낌의 앱이에요"를 확인하는 용도입니다.
    *   **특징:** 동작하는 화면 시안으로 클릭은 가능하나, 백엔드 연결이나 데이터 저장은 포함되지 않을 수 있습니다.
    *   **선택 시점:** 아이디어 구체화, 초기 사용자 피드백 수집, 내부 팀 간의 아이디어 정렬이 필요할 때. (본 강의에서 AI Studio로 만드는 것이 대표적인 프로토타입)
*   **데모 (Demo)**
    *   **목적:** 고객이나 투자자에게 "이거 됩니다!"를 보여주기 위한 발표 과정입니다.
    *   **특징:** 프로토타입이나 MVP를 가지고 발표하는 행위 자체를 의미합니다. 결과물보다는 발표 과정에 중점을 둡니다.
    *   **선택 시점:** 투자 유치, 잠재 고객 대상 제품 소개, 내부 보고 시.
*   **MVP (Minimum Viable Product)**
    *   **목적:** 핵심 기능만 넣어 시장에서 검증하고 "사람들이 진짜 쓸까?"를 테스트하는 것입니다.
    *   **특징:** 실제 사용자가 쓸 수 있는 수준으로, 로그인, 결제 등 백엔드 기능과 데이터 저장이 포함됩니다. 최소한의 기능으로 제품의 가치를 제공합니다.
    *   **선택 시점:** 시장 출시 전 사용자 반응 확인, 초기 고객 확보, 제품의 핵심 가설 검증이 필요할 때. (Lovable, Firebase Studio로 백엔드까지 구현하는 단계가 MVP)

**한눈에 흐름 정리:**
PoC → 기술 검증 | 프로토타입 → 형태 확인 | 데모 → 발표·설득 | MVP → 시장 검증

**CPO의 핵심 마인드셋:**
린 스타트업 원칙: **"빠르게 만들고 → 빠르게 검증 → 빠르게 개선"**하는 반복 과정을 통해 시장 변화에 민첩하게 대응하고, 사용자 불편함을 정확히 해결하는 데 집중합니다. "내가 쓰고 싶은 거" 대신 "사용자가 필요한 거"를 만드는 사용자 중심 디자인(UCD)이 필수입니다.

## 2. IA (정보구조) 작성 원칙과 구조

IA(Information Architecture)는 앱에 어떤 화면이 있고, 각 화면에 어떤 요소가 들어가는지 정리한 구조도입니다. CPO는 IA를 통해 제품의 전체적인 정보 흐름과 사용자 경험을 설계합니다.

**작성 원칙 (3단계):**
1.  **메인 화면 나열:** 앱에 필요한 주요 화면(탭, 메뉴 등)을 명확하게 정의합니다.
2.  **하위 요소 추가:** 각 주요 화면에 어떤 UI 요소나 기능이 포함될지 계층적으로 나열합니다.
3.  **내비게이션 결정:** 화면 간 이동 방법(하단 탭바, 햄버거 메뉴, 사이드바 등)을 결정하여 사용자 접근성을 고려합니다.

**IA 구조 (예시: 에어비앤비 클론 'StayKorea')**

```
📱 메인 (홈)
├─ 검색바 (지역, 날짜, 인원)
├─ 카테고리 필터 (해변, 산, 도시, 풀빌라...)
├─ 숙소 카드 리스트 (이미지, 이름, 가격, 별점)
└─ 하단 탭바: 탐색, 위시리스트, 여행, 프로필

🔍 숙소 상세
├─ 이미지 갤러리 (슬라이더)
├─ 숙소 정보 (이름, 설명, 호스트)
├─ 편의시설 리스트
├─ 예약 카드 (날짜 선택, 인원, 가격 계산)
└─ 리뷰 목록

❤️ 위시리스트
└─ 저장한 숙소 카드 리스트

👤 프로필
├─ 프로필 정보
└─ 로그아웃

Navigation: 하단 탭바 4개
```

## 3. UJ (User Journey) 작성 원칙과 구조

User Journey는 사용자가 서비스를 발견하고 사용하며 재방문하기까지의 전체적인 흐름을 시나리오 형태로 정리하는 것입니다. CPO는 UJ를 통해 사용자 입장에서 제품의 핵심 기능과 경험을 정의하고, 각 단계에서 필요한 기능을 도출합니다.

**작성 원칙:**
각 단계에서 사용자의 행동, 감정, 그리고 그에 따라 필요한 기능을 명확하게 기술합니다.
1.  **발견/인지:** 사용자가 문제를 느끼고 서비스의 존재를 인지하는 순간.
2.  **진입:** 앱 설치, 회원가입/로그인 등 서비스에 처음 접근하는 과정.
3.  **핵심 경험:** 서비스의 가장 중요한 기능을 처음 사용하는 순간.
4.  **일상 사용:** 사용자가 서비스를 반복적으로 이용하는 패턴.
5.  **확장/습관화:** 고급 기능을 발견하거나, 서비스가 일상에 녹아들어 재방문이 일어나는 과정.

**UJ 구조 (예시: 에어비앤비 클론 'StayKorea')**

```
1.  **발견:** "여행 가고 싶다... 숙소 찾아볼까?"
    *   **감정:** 설렘, 기대
2.  **진입:** 앱 접속 → Google 로그인
    *   **필요 기능:** 회원가입/로그인 (Auth)
3.  **탐색:** 지역/날짜 검색 → 숙소 목록 브라우징
    *   **필요 기능:** 검색, 필터, 목록 조회 (Read)
4.  **비교:** 숙소 상세 확인 → 위시리스트 저장
    *   **필요 기능:** 상세 보기, 위시리스트 (Create/Delete)
5.  **예약:** 날짜/인원 선택 → 예약 요청
    *   **필요 기능:** 예약 생성 (Create)
6.  **관리:** 예약 내역 확인 → 리뷰 작성
    *   **필요 기능:** 예약 조회, 리뷰 (Read/Create)
```

## 4. PRD (Product Requirements Document) 작성 원칙과 구조

PRD는 제품 개발의 모든 요구사항을 명세하는 핵심 기획 산출물입니다. CPO는 IA와 UJ에서 도출된 내용을 바탕으로 PRD를 작성하며, 이는 개발팀과의 커뮤니케이션과 AI 프롬프트의 원본이 됩니다.

**작성 원칙:**
*   **명확성:** 모호한 표현 없이 구체적으로 작성합니다.
*   **완결성:** 필요한 모든 정보(기능, 디자인, 기술 스택 등)를 포함합니다.
*   **가독성:** 구조화된 형태로 작성하여 쉽게 이해할 수 있도록 합니다.
*   **사용자 중심:** 사용자 문제 해결과 가치 제공에 초점을 맞춥니다.
*   **CRUD 명시:** 핵심 기능 설명 시 Create, Read, Update, Delete를 명확히 정의합니다.

**PRD 구조 (예시: 에어비앤비 클론 'StayKorea' 수준 상세화)**

```markdown
### [서비스 기획서] StayKorea (스테이코리아)

**1. 서비스명:** StayKorea (스테이코리아)

**2. 한 줄 설명:** 한국 여행자를 위한 특색 있는 숙소 검색 및 예약 플랫폼

**3. 타겟 유저:**
*   **나이:** 20-30대
*   **직업:** 국내 여행을 즐기는 직장인 및 대학생
*   **상황:** 주말여행, 단기 휴가 등 국내 숙소를 찾으며, 개인 취향을 중시하고 특별한 경험을 선호하는 사용자

**4. 핵심 가치:**
*   취향에 맞는 특색 있는 숙소를 쉽고 빠르게 발견
*   간편하고 신뢰할 수 있는 예약 경험 제공
*   여행의 설렘을 더하는 개인화된 정보 제공

**5. 핵심 기능 (Features):**
(User Journey에서 추출, CRUD 명시)

*   **① 숙소 검색 (Read):**
    *   **기능:** 지역명(예: 제주, 서울), 체크인/체크아웃 날짜, 인원수(성인/어린이/유아), 가격 범위(최소~최대)를 입력하여 숙소를 검색할 수 있습니다.
    *   **입력:** 검색바, 달력 선택, 숫자 입력 필드
    *   **출력:** 검색 조건에 맞는 숙소 목록 (숙소 카드 형태)
*   **② 숙소 목록 조회 및 필터 (Read):**
    *   **기능:** 검색 결과 및 메인 화면에서 숙소 카드 리스트를 보여줍니다.
        *   **카테고리 필터:** 해변🏖️, 산⛰️, 도시🏙️, 풀빌라🏊, 한옥🏯 등 다양한 테마 카테고리로 필터링할 수 있습니다.
        *   **정렬:** 추천순, 가격 낮은 순, 가격 높은 순, 별점 높은 순으로 정렬할 수 있습니다.
    *   **출력:** 각 숙소는 이미지, 이름, 위치(간략), 1박당 가격, 평균 별점, 리뷰 개수를 포함한 카드로 표시됩니다.
*   **③ 숙소 상세 보기 (Read):**
    *   **기능:** 특정 숙소 카드 클릭 시 상세 페이지로 이동하며, 숙소에 대한 모든 정보를 보여줍니다.
    *   **요소:**
        *   **이미지 갤러리:** 고해상도 숙소 사진 5장 이상 (슬라이더 또는 그리드 형태)
        *   **숙소 정보:** 숙소 이름, 간략 설명, 상세 설명, 호스트 정보 (프로필 사진, 이름, 응답률), 숙소 위치 (지도 연동), 편의시설 리스트 (아이콘 포함).
        *   **예약 카드:** 체크인/체크아웃 날짜 선택, 인원수 조절, 1박당 가격, 총 가격 자동 계산.
        *   **리뷰 목록:** 다른 사용자들이 남긴 리뷰 (별점, 텍스트, 작성자, 날짜).
*   **④ 위시리스트 저장/삭제 (Create/Delete):**
    *   **기능:** 숙소 카드 및 상세 페이지에서 ❤️(하트) 버튼을 토글하여 위시리스트에 숙소를 저장하거나 삭제할 수 있습니다.
    *   **결과:** 위시리스트 페이지에서 저장된 숙소 목록을 확인할 수 있습니다.
*   **⑤ 예약 요청 (Create):**
    *   **기능:** 숙소 상세 페이지의 예약 카드를 통해 체크인/체크아웃 날짜, 인원수를 선택하고 "예약하기" 버튼을 눌러 예약 요청을 보낼 수 있습니다.
    *   **프로세스:** 예약 가능 여부 확인 → 총 가격 확인 → 결제 정보 입력 (UI만 구현, 실제 결제 연동은 MVP 2단계) → 예약 완료.
*   **⑥ 리뷰 작성 (Create/Read):**
    *   **기능:** 예약 완료 후, '내 여행' 페이지에서 과거 예약 내역에 대해 별점(1-5점)과 텍스트 리뷰를 작성할 수 있습니다.
    *   **결과:** 작성된 리뷰는 해당 숙소 상세 페이지에 노출됩니다.
*   **⑦ 사용자 인증 (Auth):**
    *   **기능:** Google 계정을 통한 간편 로그인/회원가입 기능을 제공합니다.
    *   **데이터:** 사용자 프로필 정보 (이름, 프로필 사진) 저장 및 관리.

**6. 화면 목록 (Pages / Sections):**
(IA에서 추출)

*   **메인 (탐색):** 검색바 + 카테고리 필터 + 숙소 카드 그리드
*   **검색 결과:** 필터 적용된 숙소 카드 리스트

*   **내 여행:** 예약한/과거 여행 숙소 목록 + 리뷰 작성 버튼
*   **프로필:** 사용자 정보 표시 + 로그아웃 버튼
*   **네비게이션:** 하단 탭바 4개 (탐색, 위시리스트, 여행, 프로필)

**7. 디자인 가이드:**
*   **스타일:** 에어비앤비 레퍼런스, 깔끔하고 따뜻하며 직관적인 사용자 경험 제공
*   **메인 컬러:** `#FF385C` (에어비앤비 핑크)
*   **서브 컬러:** `#484848` (텍스트), `#767676` (서브 텍스트), `#EBEBEB` (경계선, 배경)

*   **접근성:** 웹 접근성 표준(WCAG) 준수
*   **언어:** 한국어 UI
```

## 5. "기획 산출물이 곧 프롬프트"라는 원칙 적용법

CPO에게 있어 기획 산출물(IA, UJ, PRD)은 더 이상 단순히 개발팀을 위한 문서가 아니라, AI에게 제품을 생성하도록 지시하는 강력한 "프롬프트"의 재료입니다. 기획서의 품질이 AI 결과물의 품질을 좌우합니다.

### 5.1. 기획 산출물과 프롬프트 요소 매핑

| 기획 산출물            | 프롬프트에서의 역할/요소           | AI Studio (UI/FE 중심)          | Lovable (CRUD/Fullstack)           | Firebase Studio (Fullstack + Auth)   |
| :--------------------- | :--------------------------------- | :------------------------------ | :--------------------------------- | :----------------------------------- |
| **IA (정보구조)**      | `Pages`, `Sections` (화면 구조)  | `Sections` 항목                 | `Pages` 항목                       | `Pages` 항목                         |
| **User Journey (흐름)** | `Features` (기능 목록)             | 주요 기능 나열 (UI 위주)        | `Features (CRUD)` 상세 명세        | 주요 기능 (Auth/DB 포함)             |
| **PRD (타겟/가치)**    | `Context` (배경 정보)              | `Context` 섹션                  | `Context` 섹션                     | `Context` 섹션                       |
| **PRD (디자인)**       | `Design System` (디자인 가이드)    | `Design` 상세                   | `Design` 상세                      | `Design` 상세                        |
| **PRD (기술 스택)**    | `Tech Stack` (기술 명세)           | `Tech` 불필요 (FE 기본)         | `Tech` 필수 명시 (DB/Auth 포함)    | `Tech` 필수 명시 (`Next.js + Firebase`) |
| **인증/DB 필요 여부**  | `Auth`/`DB` 관련 지시              | 해당 없음                       | Supabase 연동, `CRUD` 명시         | Google Auth, Firebase (Firestore) 명시 |

### 5.3. 프롬프트 품질 향상을 위한 추가 원칙 및 팁

1.  **프롬프트 엔지니어링 5원칙 적용:**
    *   **역할(Role) 부여:** AI에게 "너는 시니어 프론트엔드 개발자야"처럼 전문성을 부여하여 기대하는 결과물의 수준을 높입니다.
    *   **맥락(Context) 제공:** "누구를 위한, 어떤 상황인지" 서비스의 배경 정보를 제공하여 AI가 더 잘 이해하도록 돕습니다.
    *   **구체적 지시(Task):** 기능, 섹션, 동작을 명확하고 구체적으로 명시합니다. 모호한 표현(예: "더 예쁘게")은 피합니다.
    *   **디자인 시스템(Design):** 컬러 코드, 폰트, 레퍼런스 사이트 등을 지정하여 일관된 디자인 가이드를 제공합니다. (Stitch와 같은 디자인 AI 도구로 시안을 먼저 만들고, 그 디자인을 프롬프트에 이미지/HTML로 첨부하는 것도 효과적입니다.)
    *   **제약 조건(Constraints):** 금지사항이나 필수사항(예: "모바일 반응형 필수", "광고 없음")을 명확히 설정합니다.

2.  **CRUD(Create, Read, Update, Delete) 명확화:**
    *   모든 앱의 기본 동작인 CRUD를 프롬프트에 명확히 기술하면 AI가 기능을 정확하게 구현합니다.
    *   **CRUD 프롬프트 공식:**
        `[앱 이름]을 만들어줘. Features (CRUD): - Create: [무엇]을 [어떻게] 추가 - Read: [무엇]을 [어떤 형태]로 보기 - Update: [무엇]을 [어떻게] 수정 - Delete: [무엇]을 삭제 (+ 확인 팝업)`

3.  **수정 요청 잘 하는 법:**
    *   **한 번에 하나씩 요청:** 여러 기능을 동시에 요청하면 AI가 혼란스러워할 수 있습니다.
    *   **위치 + 속성 + 값을 명시:** "어디의 무엇을 어떻게" 변경할지 구체적으로 지시합니다. (예: "헤더의 배경색을 `#1e293b`로 바꾸고, 카드 간격을 24px로 늘려줘")
    *   **스크린샷 첨부 (Lovable 등):** 에러가 발생하면 스크린샷을 찍어 AI에게 첨부하고 설명을 요청하면 AI가 직접 디버깅하는 데 큰 도움이 됩니다.

4.  **프롬프트 자가 체크리스트:**
    *   ☐ 역할(Role)을 지정했는가?
    *   ☐ 타겟 유저를 명시했는가?
    *   ☐ 기능을 구체적으로 나열했는가? (특히 CRUD)
    *   ☐ 디자인 시스템(컬러, 폰트, 레퍼런스)을 지정했는가?
    *   ☐ 기술 스택(프레임워크, DB, 인증)을 명시했는가?
    *   ☐ 한국어 UI를 요청했는가?
    *   ☐ 샘플 데이터를 포함하도록 요청했는가?

**핵심 메시지:**
*   **"기획 없는 바이브코딩 = 예쁜 쓰레기"**: 명확한 기획 없이는 AI도 좋은 제품을 만들 수 없습니다.
*   **"프롬프트 품질 = 결과물 품질"**: 기획서가 AI 프롬프트의 핵심 재료이므로, 기획서를 얼마나 상세하고 명확하게 작성했는지가 중요합니다.

**사용성 테스트 (Usability Test):**
AI로 빠르게 프로토타입을 만들었다면, **"5명만 테스트하면 문제의 85%를 발견할 수 있습니다"** (Nielsen의 법칙). CPO는 기획-프롬프트-개발 과정 후, 반드시 사용성 테스트를 통해 제품이 의도한 대로 사용되는지 검증해야 합니다.
*   **Think Aloud 기법:** 사용자에게 "사용하면서 드는 생각을 계속 입밖으로 말해주세요"라고 요청합니다.
*   **시나리오 작성:** "원하는 식재료를 추가해보세요"처럼 답을 알려주지 않고 목표만 주세요.
