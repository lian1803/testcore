### SaaS MVP 출시 전 체크리스트 - 기술 안정성, 결제 테스트, 보안 최소 요건, 법적 요건(이용약관/개인정보처리방침/전자상거래법), 장애 대응 플랜
### **SaaS MVP 출시 전 체크리스트** (세계 최고 수준 실전 적용)

**핵심 원칙**: MVP는 **핵심 기능 3개 이하**로 제한, 출시 1주 전부터 매일 체크. 실패 사례 80%가 기술/보안 미검증에서 발생[1][2]. **총 20개 항목**, 수치 기준/자동화 위주.

#### **1. 기술 안정성 (99.9% uptime 목표, PoC 100% 통과)**
- [ ] **핵심 기능 PoC 완료**: 성능 기준(응답 <3초) 정의 후 샌드박스 테스트. 리스크 기술 A/B 테스트[1].
- [ ] **QA 병행 자동화**: 단위/회귀 테스트 CI/CD 연동, 코드 변경 시 자동 재검증. AI UI 테스트 시뮬레이션[1][2].
- [ ] **모바일/오류/성능 테스트**: 실제 기기 반응성, 3초 로드, DB 쿼리 <500ms, 더블클릭/연결 끊김 처리[2].
- [ ] **기술 부채 정리**: 코드 리뷰 100%, 롤백 플랜 정의. 초기 개발 후 20% 시간 투자[1][4].
- [ ] **배포 전략**: 스테이징 → 프로덕션, 자동화 테스트 후 릴리즈[1].

#### **2. 결제 테스트 (실제 트랜잭션 100회 성공, 0% 실패)**
- [ ] **전체 흐름 검증**: 가입 → 결제 → 취소/환불. Stripe/PayPro 등 통합, 더블클릭/오류 처리[2][5].
- [ ] **과금 모델 단순화**: 사용자/기능/사용량 중 1개 선택. 복잡 시 이탈률 30%↑[3].
- [ ] **테스트 데이터셋**: 10종 카드(성공/실패/엣지케이스), 모바일/데스크톱 50회 반복[2].
- [ ] **세션/인증 연동**: OAuth(Google/GitHub) + 이메일 인증 후 결제. 초기 버그 70% 여기서 발생[2].

#### **3. 보안 최소 요건 (OWASP Top 10 준수, 감사 100%)**
- [ ] **데이터 암호화/접근 제어**: DB 암호화(AES-256), RBAC(Role-Based Access). 초기 설계 내재화[3].
- [ ] **인증/세션 관리**: 이메일 인증, 비밀번호 초기화, JWT 토큰 만료 <24시간[2].
- [ ] **취약점 스캔**: SQL 인젝션/XSS 자동 툴(Snyk) 실행, 임계치 초과 시 블록[1][6].
- [ ] **로그/감지**: 오류 로그 → Slack 자동 알림, 서버 부하 시 autoscaling[1].

#### **4. 법적 요건 (한국 전자상거래법 필수, 1페이지 템플릿 사용)**
| 요건 | 최소 체크포인트 | 템플릿 출처 예시 |
|------|-----------------|---------------|
| **이용약관** | 환불/해지/책임 한도 명시. 구독 취소 3클릭 이내 | 정부 표준 템플릿(공정위) |
| **개인정보처리방침** | 수집/이용/제3자 제공 동의, 파기 절차. GDPR 준수 시 EU OK | 개인정보보호법 템플릿(방송통신위) |
| **전자상거래법** | 사업자 정보/통신판매 신고번호 표시, 과장광고 금지. 결제 시 동의 팝업 | 공정위 전자상거래 표준약관 |

- [ ] **페이지 내장**: 푸터 고정 링크, 모바일 보기 최적화. 변호사 리뷰 1회(비용 50만 원)[3].
- [ ] **동의 UI**: 체크박스 필수, 로그 저장(감사 대비).

#### **5. 장애 대응 플랜 (MTTR <1시간, 24/7 알림)**

### 1인~3인 팀 SaaS 프로젝트에서 자주 실패하는 원인 분석 2024 - 과도한 기능 개발, 고객 부재 상태 출시, 결제 연동 지연, 품질 미달 출시의 결과
### 1인~3인 팀 SaaS 프로젝트 실패 원인 (2024 분석)

**핵심 실패율 수치**: B2B SaaS 월 이탈률 산업 평균 3.5%, 주원인 고객 이탈(리서치/검증 부재) + 결제 실패(연동 지연).[4] 1인 솔로 팀은 검토 프로세스 부재로 AI 과의존 시 실패율 2배↑ (환각/버그 누적).[2]

#### 1. **과도한 기능 개발 (Feature Bloat)**
   - **실패 사례**: 생산성 SaaS(슬랙/노션 등) 파편화 문제 해결 위해 "통합 관리" 기능 과잉 가정 → 실제 자료 검색 실패 무시, 프로젝트 방향성 상실.[1]
   - **1-3인 팀 함정**: AI vibe coding으로 기능 쌓기 → 기술 부채 축적, 100명→10K 사용자 시 품질 깨짐 (엣지 케이스 폭증).[2]
   - **프레임워크 (MVP 우선)**: 기능 3개로 제한 출시. SUS 점수 68↑ 목표로 주 1회 사용성 테스트 (데스크 리서치+논문 분석).[1]
   - **수치**: SaaS 가격 상승 연 8-11%에도 레거시 기능 유지 시 매출 10-20% 손실.[4]

#### 2. **고객 부재 상태 출시 (No Customer Validation)**
   - **실패 사례**: 건강 습관 앱 "습관 형성 좋을 거야" 막연 전제 → 리서치 부재, 사업성 0, 사용률 저조.[1] 이벤트 호스트 인터뷰 후에도 초기 사용자 0.
   - **1-3인 팀 함정**: 솔로 창업자 가설 미검증 → AI 컨텍스트 엔지니어링 실패 (구식 아키텍처 입력 시 비생산적 출력).[2]
   - **프레임워크 (Problem-Solution Fit)**: 
     1. 데스크 리서치 (뉴스/논문) → 구체 고충 발굴 (e.g., SaaS 파편화로 자료 찾기 시간↑).[1]
     2. 설문 50명+호스트 인터뷰 → "설득 필요 시 문제 뾰족X" 기준 탈락.
     3. HCI 방법론: 근본 문제(표면 아님) 정의 후 출시.[1]
   - **수치**: 35% 팀 SaaS 자체 개발 교체 (맞춤형 우위).[5]

#### 3. **결제 연동 지연 (Payment Delay)**
   - **실패 사례**: Grandfathering(레거시 플랜)로 결제 로직 복잡 → 운영 부담↑, 이탈률 3.5% 초과.[4]
   - **1-3인 팀 함정**: 캘린더 API 연동 실패 반복 (에이전트 학습 미흡) → 출시 지연.[3]
   - **프레임워크 (Day 1 Billing)**:
     | 단계 | 액션 | 지표 |
     |------|------|------|
     | Week1 | Stripe 연동 MVP (1플랜만) | 성공률 95% |
     | Week2 | Grandfathering 피함: 신규 전환 투명 알림 | 이탈 <2% |
     | Scale | Skills-bench 벤치마크로 에이전트 학습 | API 실패 0%[3] |
   - **수치**: 결제 실패로 반복 매출 불안정, 저가 플랜 누적 손실↑.[4]

#### 4. **품질 미달 출시 (Low Quality Release)**
   - **실패 사례**: AI 생성 지원 저용량 OK → 대량 사용자 시 엣지 케이스 폭발, 지원 좌절.[2] 디자인 시스템 프로젝트도 사용률 미달.[1]
   - **1-3인 팀 함정**: 빌드/배포 인프라 불일치 → 환경 차이 버그.[3] QA 부재로 규모화 실패.
   - **프레임워크 (Scale-Proof QA)**:
     1. 동일 인프라 빌드/배포 (환경 차이 0).[3]
     2. SUS 정량 평가 + 엣지 케이스 20% 테스트.
