### 토스페이먼츠 vs 포트원(구 아임포트) 비교 2024 - 건당결제 + 월정액 구독 동시 지원, 수수료 비교, Next.js 연동 난이도, 웹훅 처리 패턴
### **토스페이먼츠 vs 포트원(구 아임포트) 비교 2024-2025 핵심**

**토스페이먼츠가 건당결제 + 월정액 구독 동시 지원에서 우위(네이티브 통합), 수수료는 거래액 5천만 원 기준 토스 우세(+25만 원/월), Next.js 연동 난이도 포트원 더 쉬움(SDK 풍부), 웹훅은 토스 큐잉 패턴 추천.**[1]

#### **1. 결제 지원 비교 (건당 + 월정액 구독 동시)**
| 항목 | 토스페이먼츠 | 포트원 |
|------|-------------|--------|
| **건당결제** | ✅ 강점 (신용/계좌/휴대/간편/해외 네이티브) | ✅ 멀티 PG 중계 (KG/이니시스 등 선택) |
| **월정액 구독** | ✅ 동시 지원 (빌링 API 내장, 자동 갱신) | ✅ 지원 (구독 API, 하지만 PG별 별도 설정) |
| **동시 운영** | **추천: 단일 PG로 간편** (토스 앱 연동 보너스) | 멀티 PG라 복잡, 스위칭 쉬움 |
| **프레임워크** | REST API + SDK (Node.js 최적) | **가장 쉬움: 10줄 코드로 멀티 PG** |

**실전 팁**: 월 1천만 원 미만 스타트업 → 포트원 (설치 0원). 5천만 원 이상 → 토스 (볼륨 할인).[1]

#### **2. 수수료 비교 (2025 기준, 설치/월 이용료 포함 TCO)**
| 결제 방식 | 토스페이먼츠 | 포트원 (KG/이니시스 중계) | **승자** |
|-----------|-------------|---------------------------|----------|
| **신용카드** | 2.3~2.7% | 2.2~2.8% (PG 따라) | 무승부 |
| **계좌이체** | **0.5~0.8%** (최저) | 0.7~1.2% | **토스** |
| **휴대폰** | 4.4~5.5% | 4.2~5.8% | 포트원 약간 |
| **간편결제** | **2.3~2.7%** | 2.4~3.2% | **토스** |
| **해외** | **3.5~4.5%** | 3.8~5.0% | **토스** |
| **월 5천만 원 사례** (카드70%+계좌30%) | **92.5만 원** (연 300만 절감) | 95만 원 | **토스 +25만/월**[1] |

**절감 전략 (실증)**:
- 계좌이체 비율 15%→35% 유도 → 수수료 2.15%→1.75% (월 20만 절감, 1% 인센티브로 고객 유치).[1]
- **TCO 계산 공식**: (거래액 × 수수료율) + 설치비 + 월료. 토스: 설치 무료/월 3만 원 추정, 포트원: 0원.[1]

#### **3. Next.js 연동 난이도 (코드 라인 수 / 학습 곡선)**
| 항목 | 토스페이먼츠 | 포트원 | **난이도 점수 (1-10, 낮을수록 쉬움)** |
|------|-------------|--------|-------------------------------------|
| **설치** | `npm i toss-payments` + API 키 | `npm i portone` + 1줄 init | 포트원 **2** vs 토스 **4** |
| **건당결제** | 20줄 (useTossPayments 훅) | **10줄** (PortOne.openPayment) | 포트원 승 |
| **구독** | 15줄 (subscribe API) | 12줄 (billing API) | 비슷 |
| **에러 핸들링** | 웹훅 필수 (큐잉 복잡) | 자동 리트라이 내장 | 포트원 **1** |

### SaaS 크레딧 시스템 설계 패턴 - 건당 차감(1크레딧=1PPT), 월정액 무제한, 크레딧 충전, 만료 정책, DB 스키마 설계
# SaaS 크레딧 시스템 설계: 실전 패턴 및 아키텍처

SaaS 크레딧 시스템은 **원장(Ledger) 기반 아키텍처**를 선택하고, 직접 잔액 수정이 아닌 **모든 차감을 거래 이력으로 기록**하는 방식이 업계 표준입니다.[3] 다음은 4가지 핵심 설계 패턴입니다.

## 1. 크레딧 차감 모델 (건당 과금)

**기본 구현 로직:**
```
1. 서비스 이용 전: 사용자 크레딧 잔액 확인
2. 필요 크레딧 충족 여부 검증
3. 서비스 제공 직후: 크레딧 차감 처리
4. 거래 기록 저장 (감시, 환불용)
```

**WordPress 예시:**[1]
- MyCred 플러그인을 통해 크레딧 시스템 구축
- 사용자별 잔액 관리: `mycred_get_users_balance($user_id)`
- 서비스 이용 시 필요 크레딧 설정: `$points_required = 5` (1PPT = 1크레딧)
- 차감 실행: `mycred_subtract("서비스명", $user_id, $points_required, '메모', time())`
- 잔액 부족 시 서비스 미제공 처리

## 2. 월정액 무제한 모델

| 요소 | 설계 방향 |
|------|---------|
| **청구 기간** | 조직 생성 시부터 시작하는 롤링 월별 청구[2] |
| **무제한 항목** | 정해진 월 구독료에 크레딧 무제한 제공 |
| **하이브리드** | 기본 무제한 + 초과분 추가 청구[6] |

**ClickHouse 구현 사례:**[2]
- 월별 청구 주기마다 신규 크레딧 자동 할당
- 미사용 크레딧 다음 월 자동 소멸 정책 적용 가능

## 3. 크레딧 충전 및 선불 시스템

**다단계 크레딧 관리:**[2]
- **평가판 크레딧**: 신규 사용자 가입 시 기본 제공 (예: 30크레딧)[1]
- **선불 크레딧**: 신용카드 결제를 통한 충전
- **약정 크레딧**: 장기 계약 시 선할인된 크레딧 패키지

**결제 연동:**[2]
- Stripe 요소 UI 사용으로 카드 정보 토큰화
- 종량제 청구 시 자동 결제 연결

## 4. 만료 정책 (크레딧 수명 관리)

**원장 설계에서의 만료 처리:**[2]
- 각 크레딧에 **시작/종료 날짜** 명시
- 청구서 생성 전 크레딧 잔여 금액 및 만료 상태 확인
- 만료된 크레딧은 자동 제외, 유효한 크레딧만

### NextAuth.js 또는 Supabase Auth 기반 사용자 인증 시스템 - 이메일/카카오/네이버 소셜 로그인, 세션 관리, 역할 기반 접근 제어(일반/프로/엔터프라이즈)
### NextAuth.js (Auth.js v5) 기반 인증 시스템 구현 (세계 최고 수준 압축)

**핵심 아키텍처**: Next.js 14+ App Router에서 `/app/api/auth/[...nextauth]/route.ts`에 NextAuth 핸들러 설정. 이메일/카카오/네이버 소셜 + JWT 세션 + Prisma 어댑터로 DB 연동 + RBAC(역할: general/pro/enterprise).[2][6]

#### 1. 설치 & 환경변수 (5분 셋업)
```
npm i next-auth @auth/prisma-adapter @prisma/client prisma
npx prisma init
```
**.env**:
```
NEXTAUTH_URL=http://localhost:3000
NEXTAUTH_SECRET=your-super-secret-key (openssl rand -base64 32 생성)
GOOGLE_CLIENT_ID=your_google_id
GOOGLE_CLIENT_SECRET=your_google_secret
KAKAO_CLIENT_ID=your_kakao_rest_api_key
KAKAO_CLIENT_SECRET=your_kakao_client_secret (Kakao Developers > 보안 발급)
NAVER_CLIENT_ID=your_naver_id
NAVER_CLIENT_SECRET=your_naver_secret
DATABASE_URL=your_postgres_url
```
카카오/네이버: Callback URL에 `http://localhost:3000/api/auth/callback/kakao` 등록 필수.[3][7]
